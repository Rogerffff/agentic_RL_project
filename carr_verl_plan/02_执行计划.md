# CaRR 在 verl Agent Loop 的执行计划

## 1. 目标与原则

目标：在 `verl` 上实现面向多跳问答 deep search 的 CaRR 风格 RL 训练，并形成可复现实验链路。  
原则：先打通工程链路，再追求论文等价；每阶段都有可量化验收。

## 2. 交付物总览

1. `CaRR` 工具适配（可在 `verl` agent loop 中稳定调用 search/open/find）。  
2. `CaRR` 奖励接入（支持 history + rubrics + forbidden_strs）。  
3. 训练配置（SFT 冷启动、RL MVP、C-GRPO 完整版）。  
4. 评估报告（主指标、工具行为指标、消融结果、失败案例）。

## 3. 分阶段计划

## Phase 0: 基线冻结与实验脚手架（1-2 天）

任务：

1. 固定基础模型、tokenizer、max turns、parser format、stop tokens。  
2. 固定训练/评测数据版本（记录 HF revision 或快照日期）。  
3. 复制并最小化现有 `verl` 多轮配置，形成专用配置目录。

验收：

1. 一条命令可启动最小多轮训练 dry-run。  
2. 配置文件中所有关键超参有注释与默认值说明。

## Phase 1: 数据准备与样本结构统一（2-3 天）

任务：

1. 将 CaRR SFT 数据转换为 `verl` 多轮对话样本格式。  
2. 将 RL 数据转换为包含 `question/answer/rubrics/forbidden_strs/metadata` 的训练样本。  
3. 构建 train/dev/test 切分并保存统计（题型、hop 数、长度分布）。

验收：

1. 数据加载通过，字段完整率 100%。  
2. 随机抽样检查 100 条，`rubrics` 与 `forbidden_strs` 不缺失。  
3. 产出数据卡（版本、来源、清洗规则）。

## Phase 2: 工具层集成（2-4 天）

任务：

1. 新增 `CaRRBrowserTool`（或 MCP 对接）并注册到 `verl` 工具系统。  
2. 映射 `search/open/find` 到统一 schema，处理超时、空结果、重试。  
3. 写工具单测与端到端冒烟测试（至少覆盖成功/失败各 1 条路径）。

验收：

1. Agent 在 3 轮内可完成“检索→打开→查找”链路。  
2. 工具调用成功率 > 95%（离线测试集）。  
3. 所有错误返回有可读错误信息，不会中断训练进程。

## Phase 3: 奖励层集成（3-5 天）

任务：

1. 扩展 agent loop 输出，确保完整 `history` 可供 reward 使用。  
2. 实现 CaRR reward 请求封装，传入 `history + rubrics + forbidden_strs + alpha`。  
3. 增加 reward trace 日志（分项分数、请求耗时、失败原因）。

验收：

1. 100 条样本 reward 请求成功率 100%。  
2. 分项奖励分布合理（无大面积常数分/NaN）。  
3. 奖励服务异常时可回退到安全默认值并继续训练。

## Phase 4: 训练策略落地（4-7 天）

任务：

1. 先做 SFT 冷启动，得到可稳定工具调用的初始策略。  
2. 做 RL MVP：标量奖励版本（正确性 + 简化 rubric）。  
3. 实现论文对齐版：C-GRPO 的 gated rubric + 组内归一化。  
4. 做小规模超参扫描（`alpha`、KL 系数、response mask、max turns）。

验收：

1. RL MVP 相比 SFT 基线在主指标上有正增益。  
2. C-GRPO 版本训练稳定，无持续性 reward collapse。  
3. 训练日志可复盘（每个 step 的关键统计齐全）。

## Phase 5: 评估与消融（3-5 天）

任务：

1. 构建主指标：EM/F1、工具调用成功率、平均轮数、token 成本。  
2. 关键消融：
   - 去掉 rubric
   - 去掉 forbidden_str 惩罚
   - 关闭 gated 逻辑
3. 输出失败案例分析（工具误用、证据不足、冗长推理等）。

验收：

1. 完整实验表格与对比曲线可复现。  
2. 每个消融结论有对应日志或样例支撑。  
3. 形成一版可分享的项目报告。

## 4. 关键技术设计建议

1. 奖励接口设计成“可替换后端”模式：本地 mock / 远端 CaRR RM 共用一套请求结构。  
2. 在样本 `extra_info` 保留 `rubrics/forbidden_strs`，避免训练中间态字段丢失。  
3. 训练前先做“轨迹完整性检查器”，阻断坏样本进入 RL。  
4. 对工具和奖励都做超时隔离，防止 worker 被长请求拖死。

## 5. 里程碑与退出条件

## M1: 工程链路打通

- 条件：工具调用 + 奖励请求都稳定，能跑完小规模训练。

## M2: 指标正向

- 条件：RL MVP 指标超过 SFT 基线，且成本可控。

## M3: 论文对齐实现

- 条件：C-GRPO 逻辑上线并完成核心消融。

若任一阶段连续两轮实验无改进，执行退出/回滚：缩小问题规模，回到上一阶段稳定配置。

## 6. 建议的目录结构（示例）

```text
carr_verl_project/
  configs/
  data/
  tools/
  rewards/
  trainers/
  scripts/
  reports/
```

## 7. 启动顺序（你可以直接照此执行）

1. 先做 Phase 0 + Phase 1，确认数据与配置完全可加载。  
2. 完成 Phase 2 后立即做端到端冒烟（不进大训练）。  
3. 完成 Phase 3 后用 1k~3k step 做 RL MVP 验证。  
4. MVP 成功再进入 Phase 4 的 C-GRPO 对齐与 Phase 5 消融。
