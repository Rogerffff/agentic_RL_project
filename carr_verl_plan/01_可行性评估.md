# 在 verl Agent Loop 框架实现 CaRR 的可行性评估

## 1. 结论

结论：**可行，但不是“开箱即用”**。  
在当前仓库状态下，可以在 `verl` 的 agent loop + reward loop 上实现 CaRR 风格训练；核心阻塞不在算法原理，而在三类工程对接：

1. `CaRR` 的 `browser.search/open/find` 工具接口与 `verl` 现有 `SearchTool` 接口不一致，需要适配层。  
2. `CaRR` 奖励服务需要完整 `history` 与 `rubrics/forbidden_strs`，需要把多轮轨迹信息从 `agent loop` 传入 `reward loop`。  
3. 论文中的 C-GRPO 逻辑（尤其 rubric 归一化与 gated 奖励）需要在 `verl` 默认 GRPO 逻辑基础上做定制。

如果你接受“先跑通 MVP，再做算法等价复现”，这个项目可以分阶段稳定推进。

## 2. 数据集完备性（按训练阶段）

| 阶段 | 是否有现成数据 | 现状判断 | 备注 |
|---|---|---|---|
| 多轮 SFT 冷启动 | 有 | 基本完整 | `CaRR` 文档说明有 832 条轨迹（由 1016 问题 reject sampling 生成） |
| RL 主训练 | 有 | 基本完整 | `CaRR` 文档说明有 2234 QA 对 + rubrics + forbidden strings |
| 验证/评测 | 部分有 | 需你补充评测协议 | 数据有，但你需要定义与项目目标一致的 dev/test 切分和指标 |

补充判断：`CaRR` 文档引用了 HF 数据源（`THU-KEG/CaRR-DeepDive`）。从“能否开展项目”的角度，数据供给是足够的；从“严格可复现论文数字”的角度，还需要核对切分、版本和预处理脚本一致性。

## 3. 工具供给完备性（给 Agent 的环境）

| 能力 | 当前供给 | 可用性评估 | 缺口 |
|---|---|---|---|
| Web 检索 / 打开页面 / 文内查找 | `CaRR/tool_server` 已提供 | 可用 | 需适配到 `verl` 工具协议（函数名、参数、返回字段） |
| 多轮工具调用解析 | `verl` 已有 `ToolParser` + `ToolAgentLoop` | 可用 | parser 当前注册格式与部分文档示例存在差异，建议固定到已支持格式 |
| 奖励计算（答案正确 + 过程质量） | `CaRR/deepsearch_rm_with_rubrics` 已提供 | 可用 | 需把 `history`、`rubrics`、`forbidden_strs` 透传给 reward server |
| 训练主流程（PPO/GRPO） | `verl/trainer/ppo` 已有 | 可用 | C-GRPO 需在奖励聚合/归一化环节定制 |

## 4. 与 verl 的关键对接点

## 4.1 工具层对接（必须做）

建议新增 `CaRRBrowserTool`（或 MCP 封装），把以下接口映射到 `verl` 的工具抽象：

- `browser.search(query)`
- `browser.open(url)`
- `browser.find(text)`

要求：

1. 返回结构与 `ToolResponse` 对齐，确保模型可读并可继续下一轮决策。  
2. 保留 `session/page` 上下文（`open/find` 依赖上下文时尤为关键）。  
3. 对失败场景（超时、空结果、非法 URL）统一错误码与可恢复提示。

## 4.2 轨迹与奖励对接（必须做）

`CaRR` 奖励服务依赖完整轨迹，不仅是最终答案。需要保证 `verl` 在每条样本上输出并保存：

- 全部多轮 `history`（assistant/tool/user）
- 本轮工具调用元信息
- `rubrics`、`forbidden_strs`、`alpha`（建议放到样本 `extra_info`）

落地方式：

1. 扩展 `agent loop` 输出字段，把完整历史写回样本。  
2. 在 `reward manager` 侧新增 CaRR 请求封装器，调用远端 RM 服务并回填分项奖励。  
3. 为调试保留 reward trace（每条样本的子分数与失败原因）。

## 4.3 C-GRPO 训练逻辑（建议分步实现）

先做两层目标：

1. **MVP**：先跑通“答案正确性 + 简化 rubric 分”的标量奖励训练。  
2. **论文等价版**：实现 gated rubric + 组内归一化的 C-GRPO 逻辑，并验证与论文公式一致。

这样可以先验证工程链路稳定，再追求算法还原。

## 5. 主要风险与缓解

1. 奖励抖动大，训练不稳定。  
缓解：先固定 `alpha` 和 reward clipping，逐步放开 rubric 权重。

2. 工具调用噪声导致 credit assignment 变差。  
缓解：先做 SFT 冷启动 + tool call 格式约束，再进入 RL。

3. `history` 丢字段导致 RM 打分无效。  
缓解：训练前对 100 条样本做“轨迹完整性单测”。

4. 文档配置与源码支持不一致（如 parser format）。  
缓解：以源码注册项为准，先做最小可运行配置并固化模板。

## 6. 最小可行路径（MVP）

1. 接入 CaRR 工具服务并完成单轮/多轮工具调用冒烟测试。  
2. 接入 CaRR 奖励服务，保证每条样本可返回结构化分数。  
3. 用小规模子集做 1k~3k step RL，验证：reward 上升、工具调用成功率稳定、答案 EM/F1 不退化。  
4. MVP 稳定后再上 C-GRPO 全量逻辑与正式规模训练。
