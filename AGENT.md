## What is verl?

verl (Volcano Engine Reinforcement Learning for LLMs) is an open-source RL post-training library for large language models. It implements the HybridFlow architecture (EuroSys 2025) which decouples computation and data dependencies to enable flexible RL dataflows (PPO, GRPO, DAPO, REINFORCE++, etc.) on top of existing LLM infrastructure. Initiated by ByteDance Seed team.

## Build & Install


## Linting & Formatting

Pre-commit handles all linting. Hooks: **ruff** (lint + format), **mypy**, **autogen-trainer-cfg** (regenerates `_generated_*.yaml` configs), **check-docstrings**, **check-license**, **compileall**.

```bash
pip install pre-commit && pre-commit install

# Run on staged files
pre-commit run

# Run on all files
pre-commit run --all-files

# Run specific hook
pre-commit run --all-files --show-diff-on-failure --color=always ruff
```

Ruff config: line-length=120, rules: E, F, UP, B, I, G. See `pyproject.toml` [tool.ruff].

mypy: globally lenient (`ignore_errors = true`) except strict on: `verl.trainer.config.algorithm`, `verl.trainer.ppo.core_algos`, `verl.trainer.ppo.reward`, `verl.workers.reward_manager.*`.

All source files must have Apache 2.0 license headers.

## Testing

### Test naming conventions
- `*_on_cpu.py` — runs on CPU only (no GPU required)
- All other test files — require GPU
- `special_distributed/` — multi-GPU tests (run via `torchrun`)
- `special_e2e/` — end-to-end training tests
- `special_sanity/` — quick sanity checks (license, docstrings, imports, structure)
- `special_npu/` — Ascend NPU tests
- `special_standalone/` — tests requiring dedicated environments

### Running tests locally

```bash
# CPU-only tests (creates a temporary pytest.ini to select *_on_cpu.py files)
echo '[pytest]\npython_files = *_on_cpu.py' > pytest.ini && pytest -s -x --asyncio-mode=auto tests/

# Single CPU test file
pytest -s -x --asyncio-mode=auto tests/utils/test_something_on_cpu.py

# GPU tests (excludes CPU-only and special tests)
pytest -s -x --ignore-glob='*on_cpu.py' --ignore-glob='*test_special_*' tests/

# Multi-GPU distributed tests (example)
torchrun --nproc_per_node=2 tests/special_distributed/test_something.py
```

CI workflows are in `.github/workflows/`. Key ones: `cpu_unit_tests.yml`, `gpu_unit_tests.yml`, `vllm.yml`, `sgl.yml`, `e2e_ppo_trainer.yml`.

## Configuration System

Hydra + OmegaConf with typed frozen dataclasses. Config files live in `verl/trainer/config/` with YAML defaults and subdirectories for each component (actor, critic, rollout, algorithm, model, engine, optim).

Entry point: `python -m verl.trainer.main_ppo` reads `verl/trainer/config/ppo_trainer.yaml` as the base config. Override any field via Hydra CLI syntax (`++key=value`).

Generated configs (`_generated_*.yaml`) are auto-generated by the `autogen-trainer-cfg` pre-commit hook from `scripts/generate_trainer_config.sh`. Do not edit them manually.

Key dataclass configs:
- `verl.trainer.config.algorithm.AlgoConfig` — RL algorithm settings (gamma, lambda, advantage estimator, KL control)
- `verl.trainer.config.config.CheckpointConfig` — checkpoint saving
- `verl.workers.config.engine.FSDPEngineConfig` / `MegatronEngineConfig` — engine-level settings
- `verl.base_config.BaseConfig` — frozen dataclass base with dict-like interface

## Architecture Overview

### Core Data Protocol: `DataProto` (`verl/protocol.py`)

The universal data exchange format between all components. Wraps:
- `batch: TensorDict` — batched PyTorch tensors
- `non_tensor_batch: dict` — NumPy arrays (strings, objects)
- `meta_info: dict` — metadata (UIDs, indices)

Supports slicing, concatenation, padding, serialization for Ray distribution. All workers accept and return `DataProto`.

### Single Controller Pattern (`verl/single_controller/`)

Ray-based distributed execution model:
- **`RayResourcePool`** — manages Ray placement groups and GPU allocation
- **`RayWorkerGroup`** — wraps a group of Ray remote workers; dispatches data, executes methods, collects results
- **`ResourcePoolManager`** — allocates resource pools to different worker types (actor, critic, reward)
- **`create_colocated_worker_cls()`** — factory to colocate multiple worker roles on the same GPU

Communication uses a dispatch-execute-collect pattern: split data across workers, invoke remote methods, aggregate results.

### Training Loop (`verl/trainer/`)

`RayPPOTrainer` (`trainer/ppo/ray_trainer.py`) orchestrates the main training loop:
1. Load batch from dataset
2. Generate rollouts (actor_rollout_wg)
3. Compute rewards (reward_manager_wg)
4. Compute values (critic_wg) — PPO only, not needed for GRPO
5. Compute advantages (GAE, GRPO, REINFORCE++, etc.)
6. Mini-batch policy/critic updates
7. Log metrics and checkpoint

Entry point: `verl/trainer/main_ppo.py` (Hydra main). Also: `main_generation.py` (generation-only), `main_eval.py` (evaluation), `sft_trainer.py` / `fsdp_sft_trainer.py` (supervised fine-tuning).

### Workers (`verl/workers/`)

Workers encapsulate model operations. The hybrid engine pattern allows a single worker to serve multiple roles:

- **`ActorRolloutRefWorker`** — can act as actor (training), rollout (inference), and/or reference policy. When `hybrid_engine=True`, all three roles share one worker.
- **`CriticWorker`** — value estimation for PPO
- **Reward Managers** (`workers/reward_manager/`) — registry-based: `NaiveRewardManager`, `BatchRewardManager`, `DAPORewardManager`, `PrimeRewardManager`

Worker implementations exist for each parallelism backend:
- `workers/fsdp_workers.py` — FSDP/FSDP2 (PyTorch native)
- `workers/megatron_workers.py` — Megatron tensor + pipeline parallel
- `workers/engine_workers.py` — generic engine-based workers

### Training Engines (`verl/workers/engine/`)

Pluggable compute backends implementing `BaseEngine`:
- **FSDP / FSDP2** — PyTorch Fully Sharded Data Parallel
- **Megatron** — Tensor + Pipeline Parallel via Megatron-LM
- **TorchTitan** — Distributed training framework
- **VeOmni** — Custom optimizer parallel
- **MindSpeed** — Ascend NPU support

### Rollout Backends (`verl/workers/rollout/`)

Inference engines for sequence generation, implementing `BaseRollout` (async interface):
- **vLLM** (`vllm_rollout/`) — high-throughput LLM serving
- **SGLang** (`sglang_rollout/`) — dynamic shape handling, multi-turn support
- **TensorRT-LLM** (`trtllm_rollout/`) — production inference
- **HF / Naive** — pure PyTorch (for testing)

Selection via `actor_rollout_ref.rollout.name` config (e.g., `vllm`, `sglang`, `hf`).

### RL Algorithms (`verl/trainer/ppo/core_algos.py`)

Core algorithm functions:
- `compute_gae_advantage_return()` — GAE (PPO)
- `compute_grpo_outcome_advantage()` — GRPO group-relative advantage
- `compute_policy_loss()` / `compute_value_loss()` — PPO losses
- `apply_kl_penalty()` — KL divergence in rewards
- `agg_loss()` — loss aggregation (token-mean, seq-mean-token-sum, etc.)
- `AdaptiveKLController` — dynamic KL penalty adjustment

Advantage estimator selection via `algorithm.adv_estimator` config: `gae`, `grpo`, `reinforce_plus_plus`, etc.

### Experimental Features (`verl/experimental/`)

- `agent_loop/` — multi-turn agentic RL with tool use
- `fully_async_policy/` — asynchronous training loops
- `one_step_off_policy/` — off-policy corrections
- `reward_loop/` — novel reward computation patterns
- `dynamic_dataset/` — curriculum learning
- `vla/` — vision-language agents

### Agent Training — Multi-Turn Agentic RL (`verl/experimental/agent_loop/`)

Server-based async rollout architecture for training LLM agents that use tools and multi-turn interactions. Uses Delta-based Tokenization to handle tokenization mismatch between chat templates and RL token-level training. Tracks `response_mask` per token (1=LLM-generated, 0=tool/interaction-inserted) so loss and advantage are only applied to model outputs.

**Key source files:**
- `agent_loop.py` — `AgentLoopBase` (abstract interface), `AsyncLLMServerManager` (load-balancing across inference servers), `AgentLoopWorker` / `AgentLoopManager` (Ray-based orchestration)
- `tool_agent_loop.py` — `ToolAgentLoop` state machine: PENDING → GENERATING → PROCESSING_TOOLS / INTERACTING → TERMINATED. `AgentData` tracks per-sample state across turns
- `single_turn_agent_loop.py` — `SingleTurnAgentLoop` for simple single-turn Q&A (default fallback)
- `tool_parser.py` — `ToolParser` registry with `HermesToolParser` (Qwen/Mistral `<tool_call>` format) and `GptOssToolParser`

**Related code:**
- Tools: `verl/experimental/agent_loop/tools/` — `BaseTool` interface, built-in tools (GSM8K calculator, Sandbox, Search, etc.), MCP integration via `MCPBaseTool`, `ToolRegistry`
- Interactions: `verl/experimental/agent_loop/interactions/` — `BaseInteraction` for multi-turn dialogue feedback (environment responses without tool calls)
- Reward: `verl/experimental/agent_loop/reward/` — `RewardLoopManager`, streaming reward during rollout, multi-layer rewards (tool-level, turn-level, trajectory-level)

## Key Patterns

- **Registry pattern** — reward managers, rollout backends, model factories, loss functions all use registries for extensibility
- **DataProto everywhere** — all inter-component communication uses `DataProto`; never pass raw tensors between workers
- **Frozen dataclass configs** — `BaseConfig` subclasses are immutable after creation; use `_mutable_fields` for exceptions
- **Pre-commit auto-generation** — `_generated_*.yaml` configs are regenerated on commit; if config dataclasses change, these must be updated (the hook handles it)

## Example: Running a Training Job

```bash
# GSM8K math with GRPO on 8 GPUs
python -m verl.trainer.main_ppo \
    data.train_files=~/data/gsm8k/train.parquet \
    data.val_files=~/data/gsm8k/test.parquet \
    data.train_batch_size=256 \
    data.max_prompt_length=512 \
    data.max_response_length=512 \
    actor_rollout_ref.model.path=Qwen/Qwen2.5-7B-Instruct \
    actor_rollout_ref.rollout.name=vllm \
    algorithm.adv_estimator=grpo \
    trainer.n_gpus_per_node=8 \
    trainer.total_epochs=15
```

Data preprocessing scripts live in `examples/data_preprocess/` (gsm8k.py, math_dataset.py, etc.). Training shell scripts with full parameter examples are in `examples/{algo}_trainer/`.
