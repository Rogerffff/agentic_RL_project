> 快速串讲 verl 的整体架构，从入口到训练循环的完整数据流，各组件角色与连接关系。

## 1. 从入口到训练：启动流程

verl 的训练入口是 `verl/trainer/main_ppo.py`，整个启动链路如下：

```
main()                          # Hydra 加载配置
  └→ run_ppo(config)            # 初始化 Ray 集群
       └→ TaskRunner.run(config) # Ray remote actor, 在 worker 节点执行
            ├→ add_actor_rollout_worker()   # 注册 Actor/Rollout Worker 类
            ├→ add_critic_worker()          # 注册 Critic Worker 类
            ├→ add_ref_policy_worker()      # 注册 Reference Policy (可选)
            ├→ init_resource_pool_mgr()     # 创建 GPU 资源池
            ├→ 加载 tokenizer/processor
            ├→ 创建 train/val dataset
            └→ RayPPOTrainer(config, ...).init_workers().fit()
```

**关键设计**：`TaskRunner` 是一个 Ray remote actor，它**不在 head 节点**运行，而是被调度到 worker 节点。这样 driver 进程（执行训练循环的进程）靠近 GPU 资源。

> 源文件：`verl/trainer/main_ppo.py:36-99` — main() 和 run_ppo()
> 源文件：`verl/trainer/main_ppo.py:108-352` — TaskRunner 类

## 3. DataProto：统一数据协议

verl 中**所有组件间的数据交换**都通过 `DataProto` 完成。它是一个 dataclass，包含三个字段：

```python
@dataclass
class DataProto:
    batch: TensorDict          # PyTorch 张量（token_ids, masks, logprobs 等）
    non_tensor_batch: dict     # NumPy 数组（字符串、对象，如 agent_name, reward_model）
    meta_info: dict            # 元数据（global_steps, timing, temperature 等）
```

**为什么要分三层？**
- `batch (TensorDict)`：所有可以 GPU 加速的张量数据。TensorDict 让你像操作单个 Tensor 一样操作整个字典——切片、拼接、padding 都是一行代码。
- `non_tensor_batch`：不能放入 Tensor 的数据（变长字符串、Python 对象）。用 NumPy object array 存储。
- `meta_info`：不随 batch 维度切分的全局信息。

**核心操作**：

| 操作 | 说明 | 用途 |
|------|------|------|
| `data[i]` | 索引单个样本，返回 DataProtoItem | 逐样本处理 |
| `data[start:stop]` | 切片 | 分 mini-batch |
| `data.chunk(N)` | 切分为 N 份 | dispatch 给 N 个 Worker |
| `DataProto.concat(list)` | 多份拼接 | collect Worker 结果 |
| `data.repeat(n, interleave=True)` | 每个样本重复 n 次 | GRPO 多 rollout |
| `data.union(other)` | 合并两个 DataProto 的字段 | 聚合各阶段计算结果 |
| `data.pop(keys)` | 弹出指定字段 | 分离生成/奖励所需数据 |

**数据在训练循环中的生命周期**：

```
Dataset → DataProto.from_single_dict(batch_dict)
  → .repeat(n) 复制 n 份用于多 rollout
  → .union(gen_output) 合并 rollout 结果
  → .union(reward) 合并奖励
  → .union(old_log_prob) 合并 log prob
  → .union(values) 合并 value (PPO)
  → compute_advantage() 计算优势
  → 分 mini-batch 送入 Actor/Critic 更新
```

> 源文件：`verl/protocol.py:328-339` — DataProto 定义

## 4. Ray Single Controller：分布式调度模式

verl 不使用 PyTorch 原生的分布式训练（如 torchrun），而是用 Ray 实现一种 **Single Controller** 模式：

### 4.1 三个核心组件

**RayResourcePool** — GPU 资源分组
```python
resource_pool_spec = {
    "global_pool": [8, 8],   # 2 个节点，每节点 8 GPU
    "reward_pool": [2],      # 1 个节点，2 GPU（可选独立奖励池）
}
```
每个 pool 对应一个 Ray Placement Group，确保 Worker 被调度到正确的 GPU 上。

**RayWorkerGroup** — Worker 集合 + 通信
```
Controller (CPU, 单进程)
     │
     │  dispatch: data.chunk(N)  → 按 DP rank 切分
     │
     ├─── Worker 0 (GPU0) ──┐
     ├─── Worker 1 (GPU1)   │  execute: remote method call
     ├─── Worker 2 (GPU2)   │
     └─── Worker 3 (GPU3) ──┘
                             │
     collect: DataProto.concat(results)  ← 聚合结果
```

这就是 **dispatch-execute-collect** 模式：
1. **dispatch**：Controller 将 DataProto 按 DP 数切分
2. **execute**：通过 Ray RPC 调用各 Worker 的方法（如 `compute_log_prob`）
3. **collect**：等待所有 Worker 完成，拼接结果

**create_colocated_worker_cls()** — 多角色共存单 GPU

```python
# 将 Actor、Critic、RefPolicy 放在同一组 GPU 上
class_dict = {
    "ActorRollout": actor_cls,
    "Critic": critic_cls,
}
worker_dict_cls = create_colocated_worker_cls(class_dict)
```

这样一个物理 Worker 进程内包含多个逻辑角色，它们**分时复用** GPU 内存（训练时用 Actor，推理时切换到 Rollout）。

> 源文件：`verl/single_controller/ray/base.py` — RayResourcePool, RayWorkerGroup
> 源文件：`verl/trainer/ppo/ray_trainer.py:672-846` — init_workers()

### 4.2 资源分配示例（8 GPU 单节点）

```
global_pool: [GPU0, GPU1, GPU2, GPU3, GPU4, GPU5, GPU6, GPU7]
             ↓     ↓     ↓     ↓     ↓     ↓     ↓     ↓
每个 GPU 上共存:
  - ActorRolloutRefWorker (训练时 = Actor, 推理时 = Rollout, LoRA 关闭时 = Ref)
  - CriticWorker (PPO only)
```

GRPO 模式（不需要 Critic）更简单：每个 GPU 只跑 ActorRolloutRefWorker。

## 5. Worker 角色与 Hybrid Engine

verl 用 `Role` 枚举定义 Worker 角色：

| Role | 说明 | 何时需要 |
|------|------|---------|
| `ActorRollout` | Actor(训练) + Rollout(推理) | 始终需要 |
| `ActorRolloutRef` | 上面 + Reference Policy | LoRA 或需要 KL 时 |
| `Critic` | Value 网络 | PPO (GRPO 不需要) |
| `RefPolicy` | 独立 Reference Policy | 非 LoRA 且需要 KL |
| `RewardModel` | 奖励模型 | 使用 DisRM/GenRM 时 |

**Hybrid Engine 的核心思想**：同一个 Worker 在不同阶段切换角色。

```
时间线：
  [Rollout 阶段]      [训练阶段]           [Rollout 阶段]
  Worker = 推理模式    Worker = 训练模式     Worker = 推理模式
  vLLM/SGLang 服务    FSDP/Megatron 训练   vLLM/SGLang 服务
  ↓                   ↓                     ↓
  sleep_replicas()    update_actor()        update_weights()
  释放 KV Cache        梯度更新              同步权重到推理引擎
```

`CheckpointEngineManager` 负责这个切换：
- `update_weights()`：将训练好的权重推送到推理引擎
- `sleep_replicas()`：让推理引擎释放 KV Cache，腾出显存给训练/奖励计算

> 源文件：`verl/trainer/ppo/utils.py` — Role 枚举
> 源文件：`verl/checkpoint_engine/` — CheckpointEngineManager

## 6. Agent 训练 vs 标准训练的关键差异

在标准（单轮）训练中，Rollout 就是一次 `generate()`。但在 Agent 训练中：

```
标准训练:
  prompt → LLM generate → response → reward → update

Agent 训练:
  prompt → LLM generate → 解析 tool call → 执行工具 → 拼接结果
        → LLM generate → 解析 tool call → 执行工具 → 拼接结果
        → LLM generate → 无 tool call → 检查交互 → 环境反馈
        → LLM generate → 结束 → reward → update
```

这带来三个关键挑战，也是 verl 架构设计的核心动机：

| 挑战 | 解决方案 | 涉及组件 |
|------|---------|---------|
| 多轮交互中 GPU 空闲（等工具返回） | 异步 Rollout：每个样本独立协程 | AgentLoopManager |
| 工具/交互 token 不应参与 loss | response_mask 区分 LLM/非 LLM token | ToolAgentLoop |
| 多轮 tokenization 不一致 | Token-based API + Delta Tokenization | AsyncLLMServerManager |

这些将在后续课程深入展开。

## 8. 本课小结

| 概念 | 一句话说明 |
|------|----------|
| **main_ppo.py → TaskRunner → RayPPOTrainer** | 三级启动链：配置加载 → Ray 调度 → 训练循环 |
| **DataProto** | 所有组件间通信的统一数据格式，三层结构 (batch/non_tensor_batch/meta_info) |
| **dispatch-execute-collect** | Driver 切分数据 → Worker 并行执行 → 聚合结果 |
| **Hybrid Engine** | 同一 Worker 在推理/训练间切换，分时复用 GPU |
| **create_colocated_worker_cls** | 多角色 Worker 共存于同一 GPU |
| **Agent 训练关键差异** | 异步多轮 + response_mask + Delta Tokenization |
