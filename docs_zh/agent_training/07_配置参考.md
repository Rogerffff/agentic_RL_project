# Agent 训练配置参考

本文档是 verl 中 Agent 训练的完整配置参考手册。涵盖从数据加载、多轮交互、rollout 推理到算法与奖励的全部配置项，并附典型配置示例。

---

## 1. 配置系统概述

verl 使用 **Hydra + OmegaConf** 配置系统，基于 typed frozen dataclasses 实现类型安全的配置管理。

- **入口命令**: `python -m verl.trainer.main_ppo`
- **基础配置文件**: `verl/trainer/config/ppo_trainer.yaml`
- **覆盖方式**: 通过 Hydra CLI 语法 `++key=value` 覆盖任意字段
- **Agent 训练示例配置**: `examples/sglang_multiturn/config/gsm8k_multiturn_grpo.yaml`

### 配置继承机制

Agent 训练的 YAML 配置文件通过 Hydra `defaults` 列表继承基础配置：

```yaml
hydra:
  searchpath:
    - file://verl/trainer/config

defaults:
  - ppo_trainer    # 继承基础 PPO 训练配置
  - _self_         # 当前文件的配置覆盖基础配置
```

基础配置 `ppo_trainer.yaml` 会进一步引入各子组件的默认配置（actor、critic、rollout、algorithm、reward 等），用户只需在自己的 YAML 中覆盖需要修改的字段。

### 配置来源

配置 dataclass 定义在以下源文件中：

| 配置模块 | 源文件路径 |
|----------|-----------|
| RolloutConfig, MultiTurnConfig, AgentLoopConfig, TraceConfig | `verl/workers/config/rollout.py` |
| ActorConfig, FSDPActorConfig | `verl/workers/config/actor.py` |
| AlgoConfig, KLControlConfig | `verl/trainer/config/algorithm.py` |
| 数据配置 (YAML) | `verl/trainer/config/data/legacy_data.yaml` |
| Rollout 配置 (YAML) | `verl/trainer/config/rollout/rollout.yaml` |

## 2. 数据配置 (data)

数据配置控制训练和验证数据的加载方式、序列长度限制以及多模态输入处理。

**YAML 默认文件**: `verl/trainer/config/data/legacy_data.yaml`

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `data.train_files` | str | `~/data/rlhf/gsm8k/train.parquet` | 训练数据 Parquet 文件路径，支持列表或单文件 |
| `data.val_files` | str | `~/data/rlhf/gsm8k/test.parquet` | 验证数据 Parquet 文件路径 |
| `data.train_batch_size` | int | 1024 | 每步训练的 batch 大小 |
| `data.max_prompt_length` | int | 512 | 最大 prompt 长度（tokens） |
| `data.max_response_length` | int | 512 | 最大 response 长度（tokens） |
| `data.return_raw_chat` | bool | True | **Agent 训练必须设为 True**，返回原始聊天消息而非 tokenized input_ids |
| `data.return_raw_input_ids` | bool | False | 是否返回未添加 chat template 的原始 input_ids |
| `data.return_full_prompt` | bool | False | 是否返回包含 chat template 的完整 prompt |
| `data.filter_overlong_prompts` | bool | False | 是否过滤超长 prompt |
| `data.filter_overlong_prompts_workers` | int | 1 | 过滤超长 prompt 的 worker 数量（大规模数据集可加速） |
| `data.truncation` | str | `"error"` | 截断模式：`error`（报错）/ `left` / `right` / `middle` |
| `data.return_multi_modal_inputs` | bool | True | 是否返回多模态输入；如果 rollout 生成新多模态输入则设为 False |
| `data.shuffle` | bool | True | 是否在 dataloader 中打乱数据 |
| `data.seed` | null/int | null | 数据打乱的随机种子 |
| `data.prompt_key` | str | `"prompt"` | 数据集中 prompt 字段名 |
| `data.reward_fn_key` | str | `"data_source"` | 数据集中用于选择奖励函数的字段名 |
| `data.train_max_samples` | int | -1 | 最大训练样本数，-1 表示使用全部 |
| `data.val_max_samples` | int | -1 | 最大验证样本数，-1 表示使用全部 |
| `data.tool_config_path` | str | null | 工具配置路径，默认引用 `multi_turn.tool_config_path` |
| `data.trust_remote_code` | bool | False | 是否允许使用远程 tokenizer 的 Python 文件 |
| `data.custom_cls.path` | str | null | 自定义 dataset 类的文件路径 |
| `data.custom_cls.name` | str | null | 自定义 dataset 类名 |

> **注意**: Agent 训练场景下，`data.return_raw_chat` 必须设为 `True`，这样多轮交互时才能正确处理聊天消息格式。通常还需将 `data.max_prompt_length` 和 `data.max_response_length` 调大（如 1024）以容纳多轮对话内容。

## 3. 多轮配置 (actor_rollout_ref.rollout.multi_turn)

多轮配置是 Agent 训练的核心，控制多轮交互的行为、工具调用格式和安全检查。

**源码**: `verl/workers/config/rollout.py` - `MultiTurnConfig`

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enable` | bool | False | **启用多轮 Agent 训练**，必须设为 True |
| `max_assistant_turns` | Optional[int] | null | 最大助手轮次，null 表示无限制（默认为 `max_length // 3`） |
| `max_user_turns` | Optional[int] | null | 最大用户/工具轮次，null 表示无限制（默认为 `max_length // 3`） |
| `tool_config_path` | Optional[str] | null | 工具配置 YAML 文件路径，定义可用工具的 schema 和实现 |
| `max_parallel_calls` | int | 1 | 单轮中最大并行工具调用数 |
| `max_tool_response_length` | int | 256 | 工具响应的最大 token 数 |
| `tool_response_truncate_side` | str | `"middle"` | 工具响应超长时的截断策略：`left` / `middle` / `right` |
| `interaction_config_path` | Optional[str] | null | 交互配置 YAML 路径（用于自定义交互逻辑） |
| `format` | str | `"hermes"` | 工具调用格式：`hermes` / `llama3_json` / `gpt-oss` / `qwen` |
| `use_inference_chat_template` | bool | False | 是否使用推理时的 chat template（True 匹配生产行为，False 使用训练时记录的 token ids） |
| `tokenization_sanity_check_mode` | str | `"strict"` | 分词一致性检查模式：`strict`（严格）/ `ignore_strippable`（忽略可去除 token）/ `disable`（禁用） |
| `num_repeat_rollouts` | Optional[int] | null | 每个样本重复 rollout 的次数 |

### 关键参数说明

- **`format`**: 不同模型使用不同的工具调用格式。Qwen 系列模型推荐使用 `hermes` 或 `qwen`，Llama 3 系列使用 `llama3_json`。
- **`use_inference_chat_template`**: 设为 True 时使用模型默认 chat template，匹配生产环境行为；设为 False 时使用训练记录的 token ids，可能包含 reasoning content 等额外内容，保证训练与 rollout 一致但会导致更长的 prompt。
- **`tokenization_sanity_check_mode`**: 多轮交互逐轮 tokenize 后拼接的结果可能与整体 tokenize 不一致。对于已知存在此问题的模型（如 Qwen/QwQ-32B、Qwen/Qwen3 系列），可设为 `disable` 以减少告警。

## 4. Agent Loop 配置 (actor_rollout_ref.rollout.agent)

Agent Loop 配置控制实验性的 agent loop 式 rollout，支持更灵活的自定义 agent 行为。

**源码**: `verl/workers/config/rollout.py` - `AgentLoopConfig`

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `num_workers` | int | 8 | 并行 AgentLoopWorker 的数量 |
| `default_agent_loop` | str | `"single_turn_agent"` | 默认 Agent 类型，当数据集中未指定 `agent_name` 时使用 |
| `agent_loop_config_path` | Optional[str] | null | 自定义 Agent 配置路径，YAML 文件中包含 AgentLoop 实例列表 |
| `agent_loop_manager_class` | Optional[str] | null | 自定义 AgentLoopManager 类的全限定名（FQN），通过 importlib 动态导入 |
| `custom_async_server.path` | Optional[str] | null | 自定义 AsyncServer 类的模块路径 |
| `custom_async_server.name` | Optional[str] | null | 自定义 AsyncServer 类名（如 `AsyncvLLMServer`） |

### Agent Loop 配置文件示例

`agent_loop_config_path` 指向的 YAML 文件格式如下：

```yaml
- name: react_agent
  _target_: recipe.langgraph_agent.react_agent_loop.ReactAgentLoop
  tools: ["get_current_temperature"]
- name: math_expression
  _target_: recipe.langgraph_agent.example.math_expression.MathExpressionReactAgentLoop
  min_terms: 2
  max_terms: 6
```

## 5. Rollout 配置 (actor_rollout_ref.rollout)

Rollout 配置控制推理引擎的选择、采样参数和并行度设置。

**源码**: `verl/workers/config/rollout.py` - `RolloutConfig`

**YAML 默认文件**: `verl/trainer/config/rollout/rollout.yaml`

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `name` | str | 必填 | 推理引擎：`vllm` / `sglang` / `hf` / `trtllm` |
| `mode` | str | `"async"` | Rollout 模式，**仅支持 `"async"`**（`"sync"` 已移除） |
| `temperature` | float | 1.0 | 采样温度 |
| `top_p` | float | 1.0 | Top-p（nucleus）采样 |
| `top_k` | int | -1 | Top-k 采样，-1 表示禁用 |
| `do_sample` | bool | True | 是否采样（False 使用 greedy 解码） |
| `n` | int | 1 | 每个 prompt 生成的 response 数量（GRPO 需 >1） |
| `repetition_penalty` | float | 1.0 | 重复惩罚系数 |
| `over_sample_rate` | float | 0.0 | 早停阈值，当完成 `(1 - over_sample_rate) * total` 个请求时中止剩余请求 |
| `prompt_length` | int | 512 | Rollout 的 prompt 长度（默认引用 `data.max_prompt_length`） |
| `response_length` | int | 512 | Rollout 的 response 长度（默认引用 `data.max_response_length`） |
| `tensor_model_parallel_size` | int | 2 | 张量并行度（TP size） |
| `data_parallel_size` | int | 1 | 数据并行度（DP size） |
| `pipeline_model_parallel_size` | int | 1 | 流水线并行度（PP size，当前 vllm/sglang/trtllm 不支持 >1） |
| `expert_parallel_size` | int | 1 | 专家并行度（EP size，MoE 模型适用） |
| `gpu_memory_utilization` | float | 0.5 | GPU 显存利用率，分配给 KV cache 的比例 |
| `max_model_len` | Optional[int] | null | Rollout 的最大模型长度 |
| `max_num_seqs` | int | 1024 | 最大并发序列数 |
| `max_num_batched_tokens` | int | 8192 | 每批最大 token 数 |
| `dtype` | str | `"bfloat16"` | 模型参数精度 |
| `enforce_eager` | bool | False | 是否禁用 CUDA graph（True 禁用） |
| `enable_chunked_prefill` | bool | True | 是否启用 chunked prefill |
| `enable_prefix_caching` | bool | True | 是否启用 prefix caching（复用 KV cache） |
| `free_cache_engine` | bool | True | 生成结束后是否释放 KV cache |
| `calculate_log_probs` | bool | False | 是否计算 rollout 的 log 概率（Rollout Correction 需设为 True） |
| `ignore_eos` | bool | False | 是否忽略 EOS token 继续生成 |
| `load_format` | str | `"dummy"` | 模型权重加载方式：`dummy`（随机初始化）/ `hf` / `safetensors` / `megatron` |
| `logprobs_mode` | Optional[str] | `"processed_logprobs"` | Log 概率模式 |
| `scheduling_policy` | Optional[str] | `"fcfs"` | 调度策略 |
| `skip_tokenizer_init` | bool | True | 是否跳过 tokenizer 初始化（假设 token in token out） |
| `multi_stage_wake_up` | bool | False | SGLang 多阶段唤醒（减少训练-rollout 切换时的峰值显存） |
| `enable_sleep_mode` | bool | True | 是否启用 sleep mode |
| `quantization` | Optional[str] | null | 量化类型：`fp8` / `torchao` |
| `quantization_config_file` | Optional[str] | null | 量化配置文件路径 |
| `enable_rollout_routing_replay` | bool | False | 是否记录 MoE 路由决策 |
| `skip_rollout` | bool | False | 是否跳过 rollout，加载之前缓存的数据 |
| `skip_dump_dir` | str | `"/tmp/rollout_dump"` | 缓存 rollout 数据的目录 |
| `sglang_engine_mode` | str | `"local"` | SGLang 引擎模式 |

### 验证采样配置 (val_kwargs)

验证阶段使用独立的采样参数：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `val_kwargs.temperature` | float | 0 | 验证时的采样温度（0 表示 greedy） |
| `val_kwargs.top_k` | int | -1 | 验证时的 Top-k |
| `val_kwargs.top_p` | float | 1.0 | 验证时的 Top-p |
| `val_kwargs.do_sample` | bool | False | 验证时是否采样 |
| `val_kwargs.n` | int | 1 | 验证时每 prompt 生成数 |

> **注意**: Agent 训练推荐使用 `sglang` 作为推理引擎（`name: sglang`），它对多轮交互和动态序列长度有更好的支持。`mode` 字段仅支持 `"async"`，`"sync"` 模式已被移除。

## 6. Trace 配置 (actor_rollout_ref.rollout.trace)

Trace 配置用于记录和可视化 rollout 过程，便于调试和监控训练质量。

**源码**: `verl/workers/config/rollout.py` - `TraceConfig`

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `backend` | Optional[str] | null | Trace 后端：`mlflow` / `weave`，null 表示禁用 |
| `token2text` | bool | False | 是否将 token id 转换为文本输出 |
| `max_samples_per_step_per_worker` | Optional[int] | null | 每步每 worker 最大 trace 样本数；null 表示全部 trace |

### Trace 样本总量计算

当设置了 `max_samples_per_step_per_worker` 时，每步的总 trace 数量为：

```
总 trace 数 = max_samples_per_step_per_worker * num_workers * n_rollouts_per_sample
```

## 8. 算法配置 (algorithm)

算法配置定义 RL 算法的核心超参数，包括 advantage estimator、KL 控制和 rollout correction。

**源码**: `verl/trainer/config/algorithm.py` - `AlgoConfig`

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `adv_estimator` | str | `"gae"` | Advantage estimator 类型：`gae` / `grpo` / `reinforce_plus_plus` 等。**Agent 训练推荐 `"grpo"`** |
| `gamma` | float | 1.0 | 折扣因子 |
| `lam` | float | 1.0 | GAE lambda 参数（bias-variance 权衡） |
| `norm_adv_by_std_in_grpo` | bool | True | GRPO 中是否按标准差归一化 advantage |
| `use_kl_in_reward` | bool | False | 是否在奖励中加入 KL 惩罚 |
| `kl_penalty` | str | `"kl"` | KL divergence 估计方法：`kl` / `abs` / `mse` / `low_var_kl` / `full` |
| `use_pf_ppo` | bool | False | 是否启用 Preference Feedback PPO |

### KL 控制配置 (algorithm.kl_ctrl)

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `type` | str | `"fixed"` | KL 控制类型：`fixed`（固定系数）/ `adaptive`（自适应） |
| `kl_coef` | float | 0.001 | 初始 KL 惩罚系数 |
| `horizon` | int | 10000 | 自适应控制器的 horizon 参数 |
| `target_kl` | float | 0.1 | 自适应控制器的目标 KL 值 |

### Filter Groups 配置 (algorithm.filter_groups)

用于 DAPO 和 Entropy 过滤的配置：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enable` | bool | False | 是否启用 filter groups |
| `metric` | Optional[str] | null | 过滤指标：`acc` / `score` / `seq_reward` / `seq_final_reward` 等 |
| `max_num_gen_batches` | int | 0 | 最大生成 batch 数，非正数表示无上限 |

### Rollout Correction 配置 (algorithm.rollout_correction)

Rollout Correction 用于处理 off-policy 问题（policy mismatch、模型过期、分布偏移）。默认为 null（禁用）。

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `rollout_is` | Optional[str] | `"sequence"` | IS 权重聚合级别：null / `"token"` / `"sequence"` |
| `rollout_is_threshold` | float | 2.0 | IS 权重截断阈值 |
| `rollout_is_batch_normalize` | bool | False | 是否对 IS 权重做 batch normalization |
| `rollout_rs` | Optional[str] | null | Rejection sampling 模式（`seq_sum_k1` / `seq_mean_k1` / `seq_mean_k3` 等） |
| `rollout_rs_threshold` | Optional[str/float] | null | Rejection sampling 阈值 |
| `bypass_mode` | bool | False | 是否使用 bypass 模式（2 个 policy 而非 3 个） |
| `loss_type` | str | `"ppo_clip"` | Bypass 模式的 loss 类型：`ppo_clip` / `reinforce` |

常用预设方法（通过 Python API 调用）：

| 预设 | 描述 |
|------|------|
| `RolloutCorrectionConfig.decoupled_token_is()` | Decoupled 模式 + Token 级 IS |
| `RolloutCorrectionConfig.decoupled_seq_is()` | Decoupled 模式 + Sequence 级 IS |
| `RolloutCorrectionConfig.bypass_ppo_clip()` | Bypass 模式 + PPO-clip |
| `RolloutCorrectionConfig.bypass_ppo_clip_geo_rs()` | Bypass 模式 + PPO-clip + Geo-RS |
| `RolloutCorrectionConfig.disabled()` | 仅记录 metrics，不应用修正 |

## 9. 奖励配置 (reward)

奖励配置定义奖励计算的方式，支持自定义奖励函数和奖励模型。

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `reward.num_workers` | int | 4 | 奖励计算 worker 数量 |
| `reward.reward_manager.name` | str | `"naive"` | 奖励管理器：`naive` / `dapo` / `limit` / `remote` / `prime` |
| `reward.reward_model.enable` | bool | False | 是否使用奖励模型 |
| `reward.reward_model.model_path` | str | null | 奖励模型路径 |
| `reward.reward_model.enable_resource_pool` | bool | False | 是否使用独立资源池（Standalone 模式） |
| `reward.custom_reward_function.path` | str | null | 自定义奖励函数的文件路径 |
| `reward.custom_reward_function.name` | str | null | 自定义奖励函数名称 |

> **注意**: Agent 训练中，奖励通常由工具执行结果（如 GSM8K 计算器的正确性）或最终任务完成度决定，通过自定义奖励函数实现。

Trainer 配置控制训练循环的行为，包括 GPU 分配、训练轮数、checkpoint 和日志。

## 13. 配置调试技巧

### 查看最终合并配置

Hydra 会将所有继承和覆盖合并为最终配置。可通过以下方式查看：

```bash
python -m verl.trainer.main_ppo --cfg job
```

### 常见配置错误

1. **`data.return_raw_chat` 未设为 True**: 多轮交互无法正确处理消息格式，导致 tokenization 错误。
2. **`rollout.mode` 设为 `"sync"`**: sync 模式已被移除，会直接报错。请使用 `"async"` 或不设置此字段。
3. **`multi_turn.enable` 为 True 但未设 `tool_config_path`**: 如果需要工具调用，必须指定工具配置文件。
4. **`adv_estimator` 使用 `"gae"` 但未配置 Critic**: GRPO 不需要 Critic，GAE（PPO）则需要。Agent 训练推荐使用 `"grpo"` 以简化配置。
5. **`max_prompt_length` 或 `max_response_length` 过小**: 多轮对话内容较长，建议设为 1024 或更大。

### 通过 CLI 覆盖配置

```bash
# 修改单个参数
python -m verl.trainer.main_ppo \
    ++data.train_batch_size=512

# 启用多轮 Agent 训练
python -m verl.trainer.main_ppo \
    ++data.return_raw_chat=True \
    ++actor_rollout_ref.rollout.name=sglang \
    ++actor_rollout_ref.rollout.multi_turn.enable=True \
    ++actor_rollout_ref.rollout.multi_turn.max_assistant_turns=5 \
    ++algorithm.adv_estimator=grpo

# 使用自定义配置文件
python -m verl.trainer.main_ppo \
    --config-path=examples/sglang_multiturn/config \
    --config-name=gsm8k_multiturn_grpo
```
