# 05 奖励系统

## 1. 奖励系统概述

verl 的奖励系统（Reward Loop）是默认的奖励计算实现，专为支持高效、灵活、易用的奖励计算而设计。核心特性：

1. **分布式奖励管理器（Distributed Reward Manager）**：通过 `RewardLoopManager` 将 batch 分片后分发给多个 `RewardLoopWorker` 并行计算，充分利用多节点资源
2. **支持混合奖励场景（规则 + 判别式 + 生成式奖励模型）**：同一训练流程中可同时使用规则奖励、判别式奖励模型（DisRM）和生成式奖励模型（GenRM）
3. **简洁可扩展的接口**：基于 registry 模式的 `RewardManagerBase` 抽象，用户可轻松注册自定义奖励管理器

---

## 2. RewardLoopManager 分布式架构

> Source: `verl/experimental/reward_loop/reward_loop.py`

### 2.1 整体架构

```
PPOTrainer
    |
    +---> AgentLoopManager.generate_sequences(batch)    # Rollout
    |
    +---> RewardLoopManager.compute_rm_score(gen_batch) # 奖励计算
            |
            +---> RewardLoopWorker #1 ----> chunk_1
            +---> RewardLoopWorker #2 ----> chunk_2
            +---> RewardLoopWorker #N ----> chunk_N
```

`RewardLoopManager` 运行在 Single Controller 上，负责创建和管理所有 `RewardLoopWorker`。训练主循环中的调用流程如下：

```python
# 初始化（在 RayPPOTrainer.init_workers 中）
from verl.experimental.reward_loop import RewardLoopManager

resource_pool = resource_pool_manager.get_resource_pool(Role.RewardModel) if use_rm else None
reward_loop_manager = RewardLoopManager(
    config=config,
    rm_resource_pool=resource_pool,
)

# Rollout
gen_batch = async_rollout_manager.generate_sequences(batch)

# 计算奖励
reward_batch = reward_loop_manager.compute_rm_score(gen_batch)
```

### 2.2 分布式计算流程

`RewardLoopManager` 将 batch 分割成小块，分发给多个 `RewardLoopWorker` 并行计算，最终汇总结果：

```python
class RewardLoopManager:
    def compute_rm_score(self, data: DataProto) -> DataProto:
        if self.reward_model_manager is not None:
            self.reward_model_manager.wake_up()

        chunks = data.chunk(len(self.reward_loop_workers))
        outputs = ray.get([
            worker.compute_score_batch.remote(chunk)
            for worker, chunk in zip(self.reward_loop_workers, chunks, strict=True)
        ])
        outputs_flat = [item for sublist in outputs for item in sublist]

        # 构造 rm_scores tensor
        scores = [item["reward_score"] for item in outputs_flat]
        # ... 构建 DataProto 返回

        if self.reward_model_manager is not None:
            self.reward_model_manager.sleep()

        return DataProto(batch=batch, non_tensor_batch=non_tensor_batch, ...)
```

### 2.3 Worker 初始化与调度

Worker 数量通过 `config.reward.num_workers` 配置。初始化时采用 Round-Robin 策略将 Worker 均匀分布到各个可用节点：

```python
def _init_reward_loop_workers(self):
    num_workers = self.config.reward.num_workers
    node_ids = [node["NodeID"] for node in ray.nodes()
                if node["Alive"] and node["Resources"].get("CPU", 0) > 0]

    for i in range(num_workers):
        node_id = node_ids[i % len(node_ids)]  # Round-robin 调度
        self.reward_loop_workers.append(
            self.reward_loop_workers_class.options(
                name=f"reward_loop_worker_{i}",
                scheduling_strategy=ray.util.scheduling_strategies.NodeAffinitySchedulingStrategy(
                    node_id=node_id, soft=True,
                ),
            ).remote(self.config, self.reward_router_address)
        )
```

## 3. Streaming Reward 机制

当满足以下条件时，奖励计算可与 Rollout 并行（流式）执行：

- 使用规则奖励（rule-based），**或**
- 奖励模型使用独立资源池（`reward.reward_model.enable_resource_pool=True`）

```python
# 在 RayPPOTrainer 中判断是否启用流式奖励
enable_agent_reward_loop = not self.use_rm or self.config.reward.reward_model.enable_resource_pool

# 将 reward_loop_workers 直接传给 AgentLoopManager
reward_loop_worker_handles = (
    self.reward_loop_manager.reward_loop_workers if enable_agent_reward_loop else None
)
self.async_rollout_manager = AgentLoopManager(
    config=self.config,
    worker_group=self.actor_rollout_wg,
    rollout_resource_pool=actor_rollout_resource_pool,
    reward_loop_worker_handles=reward_loop_worker_handles,  # 传入奖励 worker
)
```

### 3.1 流式模式工作原理

在流式模式下，`AgentLoopManager` 持有 `reward_loop_worker_handles` 的引用。每个样本 rollout 完成后，立即随机选择一个 `RewardLoopWorker` 计算奖励，无需等待整个 batch 完成：

```python
class AgentLoopWorker:
    async def _compute_score(self, output, ...):
        enable_async_reward = self.reward_loop_worker_handles is not None

        if output.reward_score is None and enable_async_reward:
            # 构造单样本 DataProto
            data = DataProto(batch=batch, non_tensor_batch=non_tensor_batch)
            # 随机选择一个 reward worker
            selected_worker = random.choice(self.reward_loop_worker_handles)
            result = await selected_worker.compute_score.remote(data)
            output.reward_score = result["reward_score"]
```

### 3.2 非流式模式

当奖励模型与 actor/rollout/reference 共享资源池（Colocate 模式）时，无法流式计算——因为 rollout 期间 GPU 被推理引擎占用。此时所有 rollout 完成后，统一调用 `RewardLoopManager.compute_rm_score()` 计算奖励。

## 4. 奖励类型

### 4.1 规则奖励（Rule-based Reward）

最简单的奖励形式，基于预定义规则计算（如字符串匹配比较模型输出与标准答案）。支持同步和异步函数，推荐使用异步函数以提升效率。

```python
async def compute_score(data_source, solution_str, ground_truth, extra_info):
    """异步自定义奖励函数"""
    if solution_matches(solution_str, ground_truth):
        return {"score": 1.0}
    return {"score": 0.0}
```

通过 `reward.custom_reward_function.path` 配置自定义奖励函数路径。

**RewardLoopWorker 的奖励计算逻辑**如下：

```python
class RewardLoopWorker:
    async def compute_score(self, data: DataProto) -> dict:
        if self.config.reward.custom_reward_function.path is not None:
            # 用户自定义奖励函数
            return await self.reward_manager.run_single(data)
        else:
            if self.config.reward.reward_model.enable:
                # 判别式奖励模型 (DisRM)
                return await self.compute_score_disrm(data)
            else:
                # 默认规则奖励
                return await self.reward_manager.run_single(data)
```

### 4.2 判别式奖励模型（DisRM）

使用判别式奖励模型（如 Skywork-Reward）对问题和模型输出进行打分。将原始 prompt 与 rollout 生成的 response 拼接后送入奖励模型，获取分类概率作为奖励分数。

通过 `reward.reward_model.model_path` 指定模型，`reward.reward_model.rollout.name` 指定推理后端（`vllm`、`sglang` 等）。

```python
class RewardLoopWorker:
    async def compute_score_disrm(self, data: DataProto) -> dict:
        disrm_prompt = await self._preprocess_reward_inputs(data)
        engine_name = self.config.reward.reward_model.rollout.name
        model_name = self.config.reward.reward_model.model_path

        if engine_name == "vllm":
            payloads = {
                "model": model_name,
                "input": disrm_prompt,
                "use_activation": False,
            }
            output = await self._post_request(payloads, "classify")
            rm_score = output["data"][-1]["probs"][-1]
        elif engine_name == "sglang":
            payloads = {
                "model": model_name,
                "input": disrm_prompt,
            }
            output = await self._post_request(payloads, "v1/embeddings")
            rm_score = output["data"][-1]["embedding"][-1]

        return {"reward_score": rm_score}
```

DisRM 不需要设置 `custom_reward_function`，系统内置了预处理和打分逻辑。

### 4.3 生成式奖励模型（GenRM）

使用生成式模型评估答案质量（如 FAPO-GenRM）。GenRM 需要同时设置：

- `reward.reward_model.model_path`：指定 GenRM 模型路径
- `reward.custom_reward_function`：指定自定义函数，负责构造 GenRM 输入 prompt、调用 GenRM 生成、解析输出为奖励分数

```python
async def compute_score_gsm8k(data_source, solution_str, ground_truth, extra_info,
                                reward_router_address, reward_model_tokenizer):
    # 1. 构造 GenRM prompt
    grm_prompt = GRM_PROMPT_TEMPLATE.format(
        problem=extra_info["question"], solution=solution_str
    )

    # 2. 调用 GenRM（通过 reward_router_address 访问奖励模型服务）
    result = await chat_complete(router_address=reward_router_address, ...)

    # 3. 解析输出
    score = int(result.choices[0].message.content.strip().split("\n\n")[-1])
    return {"score": score}
```

当启用了奖励模型（`reward.reward_model.enable=True`）时，`RewardLoopWorker` 会自动将 `reward_router_address` 和 `reward_model_tokenizer` 作为额外参数传递给自定义奖励函数，用户可在函数中通过 HTTP 请求调用奖励模型服务。

### 4.4 混合奖励（Hybrid）

可以组合以上多种奖励类型。例如在自定义奖励函数中同时使用规则判断和 GenRM 调用：

```python
async def compute_score(data_source, solution_str, ground_truth, extra_info,
                        reward_router_address=None, reward_model_tokenizer=None):
    # 规则奖励部分
    format_score = check_format(solution_str)

    # GenRM 奖励部分（如果提供了奖励模型地址）
    if reward_router_address:
        genrm_score = await call_genrm(reward_router_address, solution_str, ground_truth)
    else:
        genrm_score = 0.0

    return {"score": format_score * 0.3 + genrm_score * 0.7}
```

## 5. RewardManager 变体

> Source: `verl/experimental/reward_loop/reward_manager/`

所有 RewardManager 继承自 `RewardManagerBase`，通过 registry 模式注册和查找：

```python
from verl.experimental.reward_loop.reward_manager.base import RewardManagerBase

class RewardManagerBase(ABC):
    def __init__(self, config: DictConfig, tokenizer: AutoTokenizer, compute_score: RawRewardFn):
        self.config = config
        self.tokenizer = tokenizer
        self.compute_score = compute_score

    @abstractmethod
    async def run_single(self, data: DataProto):
        raise NotImplementedError
```

### 5.1 内置变体一览

| 名称 | 注册名 | 说明 | 适用场景 |
|------|--------|------|---------|
| `NaiveRewardManager` | `naive` | 最简单的实现，直接调用 `compute_score` 函数 | 通用场景 |
| `DAPORewardManager` | `dapo` | DAPO 算法实现，包含超长响应惩罚（overlong penalty） | DAPO 训练 |
| `RateLimitedRewardManager` | `rate_limited` | 三层速率限制（并发数、RPM、TPM），支持 timeout | 调用外部 API 有速率限制时（如 LLM-as-Judge） |
| `RemoteRewardManager` | `remote` | 在独立 Ray Worker 进程中运行奖励计算 | CPU 密集型任务（如 Math-Verify） |

### 5.2 NaiveRewardManager

最基础的实现，支持同步和异步奖励函数：

```python
@register("naive")
class NaiveRewardManager(RewardManagerBase):
    async def run_single(self, data: DataProto) -> dict:
        # 1. 提取 response、data_source、ground_truth、extra_info
        # 2. decode response_ids 为文本
        # 3. 调用 compute_score（同步或异步）
        # 4. 返回 {"reward_score": reward, "reward_extra_info": {...}}
```

### 5.3 DAPORewardManager

在 NaiveRewardManager 基础上增加了超长响应惩罚机制，当响应超过预设长度时施加线性递减惩罚：

```python
@register("dapo")
class DAPORewardManager(RewardManagerBase):
    async def run_single(self, data: DataProto) -> dict:
        # ... 基础奖励计算与 NaiveRewardManager 相同 ...

        # 超长惩罚
        if self.overlong_buffer_cfg is not None and self.overlong_buffer_cfg.enable:
            expected_len = self.max_resp_len - self.overlong_buffer_cfg.len
            exceed_len = valid_response_length - expected_len
            overlong_reward = min(
                -exceed_len / self.overlong_buffer_cfg.len * self.overlong_buffer_cfg.penalty_factor,
                0
            )
            reward += overlong_reward

        return {"reward_score": reward, "reward_extra_info": reward_extra_info}
```

配置示例：

```yaml
reward:
  reward_manager:
    name: dapo
  reward_kwargs:
    max_resp_len: 2048
    overlong_buffer_cfg:
      enable: true
      len: 256
      penalty_factor: 1.0
      log: true
```

### 5.4 RateLimitedRewardManager

为调用外部 API 的场景设计，提供三层速率限制：

1. **Concurrency limiting**（`max_concurrent`）：通过 `asyncio.Semaphore` 限制并发请求数
2. **Request rate limiting**（`max_rpm`）：通过 `AsyncTokenBucket` 限制每分钟请求数
3. **Token rate limiting**（`max_tpm`）：通过 `AsyncTokenBucket` 限制每分钟 token 数

所有 limiter 为类级别全局共享，确保跨 Worker 一致的速率控制。

```yaml
reward:
  reward_manager:
    name: rate_limited
  max_concurrent: 10
  max_rpm: 500
  max_tpm: 100000
  estimated_tokens_per_request: 2000
  timeout: 60.0
```

### 5.5 RemoteRewardManager

将奖励计算放到独立的 Ray Worker 进程中执行，避免 Python GIL 对 CPU 密集型任务的影响：

```python
@register("remote")
class RemoteRewardManager(RewardManagerBase):
    def __init__(self, ...):
        # 创建独立的 Ray Worker 进程池
        self.reward_worker = [
            RewardComputeWorker.options(
                scheduling_strategy=ray.util.scheduling_strategies.NodeAffinitySchedulingStrategy(
                    node_id=ray.get_runtime_context().get_node_id(), soft=True,
                ),
            ).remote(self.compute_score)
            for _ in range(num_reward_workers)
        ]
        self.reward_worker_pool = itertools.cycle(self.reward_worker)

    async def run_single(self, data: DataProto) -> dict:
        reward_worker = self.choose_reward_worker()
        result = await reward_worker.compute_score.remote(...)
        return {"reward_score": reward, "reward_extra_info": reward_extra_info}
```

> **注意**：`RemoteRewardManager` 不支持异步奖励函数，因为 Ray Worker 进程中无法直接运行 async 函数。

### 5.6 自定义 RewardManager

通过 registry 模式注册自定义 RewardManager：



## 6. 资源模式

奖励模型的运行方式分为两种资源模式，由 `RewardModelManager`（`verl/experimental/reward_loop/reward_model.py`）管理。

### 6.1 Colocate 模式（默认）

奖励模型与 actor/rollout/reference 共享同一资源池。

```
Colocate 模式:
  [GPU 0-7: Actor + Rollout + Ref + Reward(共享)]
```

**工作流程**：
1. 所有 rollout 完成后，推理引擎释放 GPU 缓存
2. `RewardModelManager.wake_up()` 唤醒奖励模型
3. 所有样本的奖励计算完成
4. `RewardModelManager.sleep()` 释放奖励模型资源

**优点**：无需额外 GPU 资源。
**缺点**：不支持流式奖励计算，rollout 和奖励计算串行执行。

```python
class RewardModelManager:
    def wake_up(self):
        """唤醒所有 rollout replica 实例"""
        self._run_all([replica.wake_up() for replica in self.rollout_replicas])

    def sleep(self):
        """休眠所有 rollout replica 实例"""
        self._run_all([replica.sleep() for replica in self.rollout_replicas])
```

### 6.2 Standalone 模式

奖励模型运行在独立资源池，与 actor/rollout/reference 资源隔离。

```
Standalone 模式:
  [GPU 0-5: Actor + Rollout + Ref]
  [GPU 6-7: Reward Model(独立)]
```

**配置**：

```yaml
reward:
  reward_model:
    enable: true
    enable_resource_pool: true    # 启用独立资源池
    model_path: Skywork/Skywork-Reward-Gemma-2-27B
    n_gpus_per_node: 2            # 每节点分配的 GPU 数
    nnodes: 1                     # 节点数
    rollout:
      name: vllm                  # 推理后端
      tensor_model_parallel_size: 2
```

**优点**：支持流式奖励计算，每个样本 rollout 完成后立即评估。
**缺点**：需要分配额外的 GPU 资源给奖励模型。

### 6.3 模式对比总结

| 特性 | Colocate 模式 | Standalone 模式 |
|------|--------------|----------------|
| GPU 资源 | 共享，无需额外资源 | 需要独立 GPU |
| 流式奖励 | 不支持 | 支持 |
| 计算并行度 | Rollout 与 Reward 串行 | Rollout 与 Reward 并行 |
| 配置 | `enable_resource_pool=False`（默认） | `enable_resource_pool=True` |
| 适用场景 | GPU 资源紧张 | 追求训练吞吐量 |

## 7. Agent 训练中的多层奖励

Agent 训练中存在多个层次的奖励信号，它们在不同阶段产生并最终用于优势估计（advantage estimation）。

### 7.1 工具奖励（Tool Reward）

每次工具调用返回的步骤级奖励。

- **来源**：`BaseTool.execute()` 返回的 reward 值
- **存储**：`extra_fields["tool_rewards"]`（列表，每个元素对应一次工具调用的奖励）

```python
# 在 ToolAgentLoop 中收集
if tool_reward is not None:
    agent_data.tool_rewards.append(tool_reward)
```

### 7.2 轮次评分（Turn Score）

每轮交互结束时的评分。

- **来源**：`BaseInteraction.generate_response()` 返回的 score 值
- **存储**：`extra_fields["turn_scores"]`（列表，每个元素对应一轮交互的评分）

```python
# 在 ToolAgentLoop 中收集
if reward is not None:
    agent_data.turn_scores.append(reward)
```

### 7.3 轨迹奖励（Trajectory Reward）

整条轨迹的最终奖励，由 `RewardLoopManager` 计算。

- **来源**：规则奖励、DisRM 或 GenRM 计算结果
- **存储**：`DataProto.batch["rm_scores"]`

### 7.4 奖励信号的汇总

在 `NaiveRewardManager.run_single()` 中，步骤级奖励（`tool_rewards`）和轮次评分（`turn_scores`）通过 `extra_info` 传递给自定义奖励函数，用户可以在奖励函数中综合利用这些信号：

```python
# NaiveRewardManager 中的处理
num_turns = data_item.non_tensor_batch.get("__num_turns__", None)
rollout_reward_scores = data_item.non_tensor_batch.get("reward_scores", {})
extra_info["num_turns"] = num_turns
extra_info["rollout_reward_scores"] = rollout_reward_scores
```

这些多层奖励信号最终在优势估计（如 GRPO、GAE）中综合使用，指导策略优化。

## 8. 自定义奖励函数编写指南

### 8.1 函数签名

奖励函数必须遵循以下签名（同步或异步均可）：

```python
def compute_score(data_source, solution_str, ground_truth, extra_info, **kwargs) -> dict | float:
    ...
```

或异步版本：

```python
async def compute_score(data_source, solution_str, ground_truth, extra_info, **kwargs) -> dict | float:
    ...
```

**参数说明**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `data_source` | `str` | 数据来源标识 |
| `solution_str` | `str` | 模型生成的响应文本（已 decode） |
| `ground_truth` | `Any` | 标准答案（来自数据集的 `reward_model.ground_truth` 字段） |
| `extra_info` | `dict` | 额外信息，包含 `num_turns`、`rollout_reward_scores` 等 |
| `reward_router_address` | `str`（可选） | 奖励模型服务地址（启用奖励模型时自动传入） |
| `reward_model_tokenizer` | `AutoTokenizer`（可选） | 奖励模型的 tokenizer（启用奖励模型时自动传入） |

**返回值**：
- `dict`：必须包含 `"score"` 键，可附加其他键作为 `reward_extra_info`
- `float`：直接作为奖励分数

### 8.2 异步奖励函数示例（推荐）

```python
import aiohttp

async def compute_score(data_source, solution_str, ground_truth, extra_info):
    """异步自定义奖励函数 - 可以调用外部 API"""
    api_url = "http://my-reward-api/score"
    payload = {"solution": solution_str, "answer": ground_truth}

    async with aiohttp.ClientSession() as session:
        async with session.post(api_url, json=payload) as resp:
            result = await resp.json()

    score = result["score"]
    return {"score": score, "detail": result.get("detail", "")}
```

### 8.3 同步奖励函数示例

```python
def compute_score(data_source, solution_str, ground_truth, extra_info):
    """同步自定义奖励函数"""
    if answer_matches(solution_str, ground_truth):
        return {"score": 1.0}
    return {"score": 0.0}
```

### 8.4 GenRM 奖励函数示例

```python
async def compute_score(data_source, solution_str, ground_truth, extra_info,
                        reward_router_address, reward_model_tokenizer):
    """通过 GenRM 计算奖励"""
    # 1. 构造 GenRM prompt
    grm_prompt = f"评估以下解答的正确性：\n问题：{extra_info['question']}\n解答：{solution_str}"

    # 2. 调用 GenRM 服务
    result = await chat_complete(
        router_address=reward_router_address,
        messages=[{"role": "user", "content": grm_prompt}],
    )

    # 3. 解析输出
    response_text = result.choices[0].message.content.strip()
    score = parse_genrm_output(response_text)
    return {"score": score}
```

### 8.5 配置方式

```yaml
reward:
  custom_reward_function:
    path: "path/to/my_reward.py"    # 奖励函数文件路径
    name: "compute_score"           # 函数名
    reward_kwargs:                  # 额外参数（可选，会 merge 到函数调用中）
      key1: value1
      key2: value2
```

`reward_kwargs` 中的参数会通过 `partial` 自动合并到奖励函数的调用参数中，用户可以在奖励函数签名中添加对应参数来接收。
