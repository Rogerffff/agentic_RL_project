# 工具系统

## 1. 工具系统概述

verl 的工具系统允许 Agent 在多轮对话中调用外部工具（计算器、搜索引擎、代码沙箱等），工具返回的结果作为新的 context 继续对话。工具系统采用注册模式（Registry Pattern），支持两种类型的工具：

- **Native 工具**：直接在 Python 进程内执行的本地工具，如数学答案验证、代码沙箱执行等。
- **MCP (Model Context Protocol) 工具**：通过标准化协议与外部工具服务通信的远程工具，支持 SSE 和 STDIO 两种传输方式。

工具系统的核心源码位于 `verl/tools/` 目录下，主要包含以下组件：

| 文件 | 说明 |
|------|------|
| `base_tool.py` | 工具基类，定义统一的生命周期接口 |
| `schemas.py` | 数据结构定义（ToolResponse、OpenAI Schema 等） |
| `mcp_base_tool.py` | MCP 工具基类，封装 MCP 协议通信 |
| `utils/tool_registry.py` | 工具注册与加载机制 |

## 2. BaseTool 接口设计

**源码位置**: `verl/tools/base_tool.py`

`BaseTool` 是所有工具的基类，定义了工具的统一生命周期接口。每个工具都需要实现以下方法：

```python
class BaseTool:
    def __init__(self, config: dict, tool_schema: OpenAIFunctionToolSchema):
        self.config = config
        self.tool_schema = tool_schema or self.get_openai_tool_schema()
        assert self.tool_schema is not None, "Tool schema is not set!"
        self.name = self.tool_schema.function.name

    def get_openai_tool_schema(self) -> OpenAIFunctionToolSchema:
        """返回 OpenAI 格式的工具 Schema"""
        return self.tool_schema

    async def create(self, instance_id=None, **kwargs) -> tuple[str, ToolResponse]:
        """创建工具实例（每条轨迹一个实例）"""

    async def execute(self, instance_id, parameters, **kwargs) -> tuple[ToolResponse, float, dict]:
        """执行工具，返回 (响应, 步骤奖励, 指标)"""

    async def calc_reward(self, instance_id, **kwargs) -> float:
        """计算工具状态对应的奖励"""

    async def release(self, instance_id, **kwargs) -> None:
        """释放工具实例资源"""
```

### 工具生命周期

工具实例的完整生命周期如下：

```
create() ------> execute() ------> execute() ------> ... ------> calc_reward() ------> release()
   |                 |                 |                              |                    |
   |            可多次调用         可多次调用                     计算最终奖励           清理资源
   |
   创建实例,
   分配 instance_id
```

各方法的职责说明：

- **`create()`**：为每条训练轨迹（trajectory）创建一个独立的工具实例。如果未提供 `instance_id`，会自动生成 UUID。可以通过 `kwargs` 传入初始化参数（如 `ground_truth`）。返回 `(instance_id, ToolResponse)` 元组。
- **`execute()`**：执行工具的核心逻辑。接收 `instance_id` 和 `parameters`（从模型的 function call 中解析得到的参数字典）。返回三元组 `(ToolResponse, float, dict)`，分别是工具响应、步骤奖励（step reward）和指标（metrics）。该方法被 `@rollout_trace_op` 装饰器包裹，用于 rollout tracing。
- **`calc_reward()`**：根据工具当前状态计算奖励值。通常在轨迹结束时调用，用于计算最终奖励。
- **`release()`**：释放工具实例占用的资源，清理内部状态字典。

## 3. ToolResponse 数据结构

**源码位置**: `verl/tools/schemas.py`

`ToolResponse` 是工具执行结果的统一封装，基于 Pydantic `BaseModel` 实现，支持多模态返回：

```python
class ToolResponse(BaseModel):
    text: str | None = None        # 文本输出
    image: list[Any] | None = None # 图片列表（PIL Image 等）
    video: list[Any] | None = None # 视频列表

    def is_empty(self) -> bool:
        """判断响应是否为空"""
        return not self.text and not self.image and not self.video

    def is_text_only(self) -> bool:
        """判断是否仅包含文本"""
        return self.text and not self.image and not self.video
```

使用注意事项：

- **图片和视频必须以列表形式传入**。`ToolResponse` 内置了 Pydantic `model_validator`，如果传入非列表类型的 `image` 或 `video`，会抛出 `ValueError`。
- 对于单张图片，应使用 `ToolResponse(image=[img])` 而非 `ToolResponse(image=img)`。
- 图片和视频需要在返回前用对应的 `process_image` / `process_video` 方法处理，以适配多模态模型的输入格式。

## 4. OpenAI 工具 Schema

**源码位置**: `verl/tools/schemas.py`

verl 采用 OpenAI Function Calling 格式定义工具的 Schema，包含以下层级结构：

```python
class OpenAIFunctionToolSchema(BaseModel):
    """顶层工具 Schema"""
    type: str                        # 固定为 "function"
    function: OpenAIFunctionSchema

class OpenAIFunctionSchema(BaseModel):
    """函数定义"""
    name: str                        # 工具名称
    description: str                 # 工具描述
    parameters: OpenAIFunctionParametersSchema  # 参数定义
    strict: bool = False             # 是否启用严格模式

class OpenAIFunctionParametersSchema(BaseModel):
    """参数定义"""
    type: str                        # 通常为 "object"
    properties: dict[str, OpenAIFunctionPropertySchema]  # 参数属性
    required: list[str]              # 必填参数列表

class OpenAIFunctionPropertySchema(BaseModel):
    """单个参数属性"""
    type: str                        # 参数类型（string, number, array 等）
    description: str | None = None   # 参数描述
    enum: list[str] | None = None    # 枚举值（可选）
```

此外，`schemas.py` 还定义了用于解析模型输出的辅助类：

- **`OpenAIFunctionParsedSchema`**：模型输出的原始 function call（`arguments` 为 JSON 字符串）。
- **`OpenAIFunctionCallSchema`**：解析后的 function call（`arguments` 为字典），提供 `from_openai_function_parsed_schema()` 静态方法进行安全的 JSON 解析，处理 `JSONDecodeError` 等异常。
- **`OpenAIFunctionToolCall`**：完整的工具调用对象，包含 `id`、`type` 和 `function` 字段。

## 5. 工具配置 YAML 格式

工具通过 YAML 配置文件定义，在训练配置中通过 `actor_rollout_ref.rollout.multi_turn.tool_config_path` 指定路径。

### Native 工具配置示例

**示例文件**: `examples/sglang_multiturn/config/tool_config/gsm8k_tool_config.yaml`

```yaml
tools:
  - class_name: "verl.tools.gsm8k_tool.Gsm8kTool"    # 工具类的全限定名
    config:
      type: native                                      # 工具类型：native 或 mcp
    tool_schema:                                         # OpenAI 格式的工具定义
      type: "function"
      function:
        name: "calc_gsm8k_reward"
        description: "A tool for calculating the reward of gsm8k. (1.0 if parsed answer is correct, 0.0 if parsed answer is incorrect or not correctly parsed)"
        parameters:
          type: "object"
          properties:
            answer:
              type: "string"
              description: "The model's answer to the GSM8K math problem, must be a digits"
          required: ["answer"]
```

### MCP 工具配置示例

**示例文件**: `examples/sglang_multiturn/config/tool_config/mcp_tool_config.yaml`

```yaml
tools:
  - class_name: verl.tools.mcp_search_tool.MCPSearchTool
    config:
      rate_limit: 120
      timeout: 120
      type: mcp
    mcp:
      mcp_servers_config_path: ./mcp_server.json      # MCP 服务器配置文件路径
      tool_selected_list:                               # 可选：指定使用哪些工具
        - tavily_search_tool
```

### 配置字段说明

| 字段 | 说明 |
|------|------|
| `class_name` | 工具类的 Python 全限定名（module.ClassName），由 `tool_registry` 动态导入 |
| `config.type` | 工具类型，`native` 或 `mcp` |
| `tool_schema` | Native 工具的 OpenAI Function Schema 定义 |
| `mcp` | MCP 工具的额外配置（服务器地址、工具选择等） |

## 6. 内置工具介绍

### GSM8K Tool

**源码位置**: `verl/tools/gsm8k_tool.py`

数学答案验证工具，用于 GSM8K 数学问题训练场景。

- **`create()`**：接收 `ground_truth` 参数，保存标准答案。
- **`execute()`**：接收模型生成的答案，与标准答案进行比较，计算奖励分数。如果答案未添加 `#### ` 前缀，会自动补充。同时对未改进的重复提交施加 -0.05 的惩罚奖励。
- **`calc_reward()`**：使用 `verl.utils.reward_score.gsm8k.compute_score()` 计算分数，正确返回 1.0，错误返回 0.0。

### Geo3K Tool

**源码位置**: `verl/tools/geo3k_tool.py`

几何题答案验证工具，用于 Geo3K 几何问题训练场景。设计与 GSM8K Tool 类似，使用 `verl.utils.reward_score.geo3k.compute_score()` 计算分数。

### Search Tool

**源码位置**: `verl/tools/search_tool.py`

搜索工具，支持本地 Dense Retrieval 搜索服务。

- 通过 Ray Actor 实现并发执行和全局速率限制（Token Bucket 算法）。
- 可配置 `retrieval_service_url`（检索服务地址）、`num_workers`（并发数）、`rate_limit`（速率限制）、`topk`（返回结果数）等参数。
- `execute()` 接收 `query_list` 参数（查询列表），调用外部检索服务获取结果。

配置示例（`examples/sglang_multiturn/config/tool_config/search_tool_config.yaml`）：

```yaml
tools:
  - class_name: verl.tools.search_tool.SearchTool
    config:
      retrieval_service_url: http://127.0.0.1:8000/retrieve
      num_workers: 120
      rate_limit: 120
      timeout: 30
      type: native
    tool_schema:
      type: function
      function:
        name: search
        description: Searches the web for relevant information based on the given query.
        parameters:
          type: object
          properties:
            query_list:
              type: array
              item:
                type: string
              description: A list of fully-formed semantic queries.
          required:
            - query_list
```

### Sandbox Fusion Tool

**源码位置**: `verl/tools/sandbox_fusion_tools.py`

代码沙箱执行工具，支持在安全沙箱环境中执行代码。

### Image Zoom In Tool

**源码位置**: `verl/tools/image_zoom_in_tool.py`

图片区域放大工具，支持多模态 Agent 的图片分析场景。

- **`create()`**：接收 `image` 参数，支持多种图片格式（PIL Image 对象、HTTP URL、本地文件路径、base64 编码字符串）。
- **`execute()`**：接收 `bbox_2d`（四元素列表 `[x1, y1, x2, y2]`）和可选的 `label` 参数，裁剪并返回指定区域的图片。
- 内置 bounding box 验证和自动调整机制，确保裁剪区域不小于 `MIN_DIMENSION`（28 像素），超出图片边界时自动修正。
- 返回 `ToolResponse(image=[cropped_image], text=...)`，同时包含裁剪后的图片和文本描述。

## 7. MCP 工具集成

**源码位置**: `verl/tools/mcp_base_tool.py`

MCP (Model Context Protocol) 是一个标准化协议，允许通过统一接口集成各种外部工具服务。verl 提供了 `MCPBaseTool` 基类来简化 MCP 工具的开发。

### MCPBaseTool 基类

`MCPBaseTool` 继承自 `BaseTool`，封装了 MCP 协议通信的通用逻辑：

```python
class MCPBaseTool(BaseTool):
    def __init__(self, config: dict, tool_schema: OpenAIFunctionToolSchema):
        super().__init__(config, tool_schema)
        self._instance_dict = {}
        self.timeout = config.get("timeout", 30)

    async def _call_tool(self, instance_id, parameters) -> tuple[str, dict]:
        """调用 MCP 服务器执行工具，处理异常"""
        call_tool_result = await ClientManager.call_tool(self.name, parameters, self.timeout)
        result, metadata = self._parse_tool_result(call_tool_result.content)
        return result, metadata

    def _parse_tool_result(self, content: list) -> tuple[str, dict]:
        """解析 MCP 服务器返回的内容，默认拼接所有 text 类型的内容"""
        tools_content = [part.text for part in filter(lambda x: x.type == "text", content)]
        return " ".join(tools_content), {}
```

### MCP 服务器配置

MCP 服务器通过 JSON 配置文件定义，支持两种传输方式：

```json
{
    "mcpServers": {
        "SSE Server": {
            "url": "your_server_url",
            "auth_token": "your_token"
        },
        "STDIO Server": {
            "command": "npx",
            "args": ["-y", "server-mcp@0.2.1"]
        }
    }
}
```

- **SSE (Server-Sent Events)**：通过 HTTP 长连接通信，适合远程服务。需要提供 `url` 和可选的 `auth_token`。
- **STDIO**：通过标准输入输出通信，适合本地进程。需要提供 `command` 和 `args`。

### ClientManager

**源码位置**: `verl/tools/utils/mcp_clients/McpClientManager.py`

`ClientManager` 是一个全局单例对象，负责管理所有 MCP 客户端连接：

- **`initialize()`**：根据配置文件初始化所有 MCP 客户端连接，并创建 Token Bucket 速率限制器。
- **`fetch_tool_schemas()`**：从所有已连接的 MCP 服务器获取工具的 OpenAI Schema 定义，支持通过 `tool_selected_list` 过滤。
- **`call_tool()`**：根据工具名称路由到对应的 MCP 客户端并执行调用，内置速率限制。

### 自定义 MCP 工具

继承 `MCPBaseTool` 并重写 `_parse_tool_result()` 方法即可自定义 MCP 工具的结果解析逻辑。以 `MCPSearchTool` 为例（`verl/tools/mcp_search_tool.py`）：

```python
class MCPSearchTool(MCPBaseTool):
    def _parse_tool_result(self, content: list) -> tuple[str, dict]:
        """自定义解析 MCP 搜索服务器返回的内容"""
        res = ""
        res_cnt = 0
        query_list = []
        metadata = {
            "api_request_error": "",
            "status": "unknown",
            "total_results": 0,
        }
        for part in content:
            if part.type != "text":
                continue
            # 解析 query 和 results
            text = part.text.replace("'", '"')
            query_match = re.search(r'query"\s*:\s*"([^"]+)"', text)
            query = query_match.group(1) if query_match else ""
            query_list.append(query)
            # ... 解析搜索结果
        metadata["status"] = "success"
        metadata["queries"] = query_list
        metadata["query_count"] = len(query_list)
        metadata["total_results"] = res_cnt
        return res, metadata
```

## 8. ToolRegistry 注册机制

**源码位置**: `verl/tools/utils/tool_registry.py`

`ToolRegistry` 提供从 YAML 配置文件动态加载工具的能力，是工具系统的入口。

### 核心函数

```python
from verl.tools.utils.tool_registry import initialize_tools_from_config

# 从配置文件加载所有工具
tool_list = initialize_tools_from_config("path/to/tool_config.yaml")
# 构建工具名称到工具实例的映射
tools = {tool.name: tool for tool in tool_list}
```

### 加载流程

`initialize_tools_from_config()` 的内部流程如下：

1. 使用 OmegaConf 加载 YAML 配置文件。
2. 遍历配置中的每个工具定义。
3. 通过 `get_tool_class()` 使用全限定类名（如 `verl.tools.gsm8k_tool.Gsm8kTool`）动态导入工具类。
4. 根据 `config.type` 区分工具类型：
   - **`native`**：直接解析 `tool_schema` 并实例化工具类。
   - **`mcp`**：创建临时事件循环，异步初始化 MCP 客户端（连接服务器、获取工具 Schema），然后为每个 MCP 工具创建实例。
5. MCP 工具初始化时会进行最多 10 次重试（每次间隔 2 秒），确保 MCP 客户端就绪。
6. 清理临时事件循环，返回工具列表。

### 工具类型枚举

```python
class ToolType(Enum):
    NATIVE = "native"  # 本地 Python 工具
    MCP = "mcp"        # MCP 协议远程工具
```

## 9. 工具响应截断策略

**源码位置**: `verl/experimental/agent_loop/tool_agent_loop.py`
**配置类**: `verl/workers/config/rollout.py` (`MultiTurnConfig`)

当工具返回的文本内容过长时，系统会根据配置进行截断，以控制 token 消耗。

### 相关配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `actor_rollout_ref.rollout.multi_turn.max_tool_response_length` | 256 | 工具响应的最大字符长度 |
| `actor_rollout_ref.rollout.multi_turn.tool_response_truncate_side` | `middle` | 截断策略 |

### 三种截断策略

当 `len(tool_response_text) > max_tool_response_length` 时触发截断：

- **`left`**：保留前面的内容，截断末尾。

  ```
  content[:max_len] + "...(truncated)"
  ```

- **`right`**：保留末尾的内容，截断开头。

  ```
  "(truncated)..." + content[-max_len:]
  ```

- **`middle`**（默认）：保留首尾，截断中间部分。

  ```
  content[:half] + "...(truncated)..." + content[-half:]
  ```

  其中 `half = max_tool_response_length // 2`。

截断策略的选择取决于工具返回内容的特点：
- 搜索结果通常选择 `left` 或 `middle`，因为前面的结果通常更相关。
- 代码执行结果可能选择 `right`，因为错误信息通常出现在末尾。
- 默认的 `middle` 策略在大多数场景下表现良好，同时保留了上下文的开头和结尾信息。

## 10. 自定义工具开发指南

开发自定义工具需要以下步骤：

### 步骤 1：继承 BaseTool，实现核心方法

```python
from typing import Any, Optional
from uuid import uuid4

from verl.tools.base_tool import BaseTool
from verl.tools.schemas import OpenAIFunctionToolSchema, ToolResponse


class MyCustomTool(BaseTool):
    def __init__(self, config: dict, tool_schema: OpenAIFunctionToolSchema):
        super().__init__(config, tool_schema)
        self._instance_dict = {}

    async def create(self, instance_id: Optional[str] = None, **kwargs) -> tuple[str, ToolResponse]:
        """创建工具实例，初始化状态"""
        if instance_id is None:
            instance_id = str(uuid4())
        # 从 kwargs 中获取初始化参数
        create_kwargs = kwargs.get("create_kwargs", {})
        self._instance_dict[instance_id] = {
            "state": create_kwargs,
            "reward": 0.0,
        }
        return instance_id, ToolResponse()

    async def execute(self, instance_id: str, parameters: dict[str, Any], **kwargs) -> tuple[ToolResponse, float, dict]:
        """执行工具逻辑"""
        input_text = parameters.get("input", "")
        # 执行自定义逻辑
        result = self._process(input_text)
        reward = 1.0 if self._is_correct(result) else 0.0
        self._instance_dict[instance_id]["reward"] = reward
        return ToolResponse(text=str(result)), reward, {}

    async def calc_reward(self, instance_id: str, **kwargs) -> float:
        """返回累积奖励"""
        return self._instance_dict[instance_id]["reward"]

    async def release(self, instance_id: str, **kwargs) -> None:
        """清理实例"""
        if instance_id in self._instance_dict:
            del self._instance_dict[instance_id]

    def _process(self, input_text: str) -> str:
        # 自定义处理逻辑
        return input_text

    def _is_correct(self, result: str) -> bool:
        # 自定义验证逻辑
        return len(result) > 0
```

### 步骤 2：编写工具配置 YAML

```yaml
tools:
  - class_name: "my_module.MyCustomTool"
    config:
      type: native
      # 其他自定义配置项
      custom_param: "value"
    tool_schema:
      type: "function"
      function:
        name: "my_tool"
        description: "My custom tool description"
        parameters:
          type: "object"
          properties:
            input:
              type: "string"
              description: "Input text to process"
          required: ["input"]
```

### 步骤 3：在训练配置中指定 tool_config_path

通过 Hydra CLI 覆盖或直接修改训练配置：


### 开发注意事项

1. **所有方法都是异步的**：`create()`、`execute()`、`calc_reward()`、`release()` 均为 `async` 方法，在实现中可以使用 `await`。
2. **instance_id 隔离**：每条训练轨迹对应一个独立的 `instance_id`，工具内部应通过 `_instance_dict` 等方式维护各实例的独立状态，避免状态串扰。
3. **资源清理**：务必在 `release()` 中正确清理资源，避免内存泄漏。训练过程中会并发创建大量工具实例。
4. **异常处理**：`execute()` 方法中应妥善处理异常，返回包含错误信息的 `ToolResponse`，而非抛出异常导致训练中断。
5. **步骤奖励设计**：`execute()` 返回的 `float` 是步骤级奖励（step reward），可用于引导模型的工具使用行为。例如，对重复提交未改进答案施加负奖励。
6. **config 传递**：YAML 中 `config` 下的所有字段都会作为字典传入 `__init__()` 的 `config` 参数中，可用于配置工具的行为（如服务地址、超时时间、速率限制等）。
