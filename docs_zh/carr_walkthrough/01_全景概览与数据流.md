# Part 1: 全景概览与数据流

## 1.1 项目目标

我们要做的事情用一句话概括：**在 verl 框架下，训练一个能够多轮使用浏览器工具进行深度搜索的 LLM Agent，使其能回答复杂的多跳问答问题。**

具体来说：
- **基座模型**: Qwen3-4B（dense 4B 参数）
- **训练框架**: verl（字节的开源 RL post-training 库）
- **训练流程**: SFT 冷启动 → RL（C-GRPO 算法）微调
- **奖励框架**: CaRR（Citation-Aware Rubric Rewards）
- **硬件**: 8 GPU × 1 节点

## 1.2 为什么需要 CaRR？—— 核心问题与解决方案

### 传统 RL 训练 Deep Search Agent 的痛点

传统方法使用**二元结果奖励**（答对=1，答错=0）来训练。这导致两个严重问题：

**问题 1: 捷径利用（Shortcut Exploitation）**
模型发现可以跳过中间搜索步骤，直接猜答案。训练后搜索步数反而减少，因为"不搜索直接猜"有时也能猜对，获得奖励。

**问题 2: 幻觉（Hallucination）**
模型编造参考文献。即使引用了 URL，内容可能完全是捏造的，但因为最终答案碰巧正确，仍然获得满分奖励。

### CaRR 的解决方案

CaRR 提出了**细粒度的过程奖励**，不仅看结果对不对，还看推理过程好不好：

```
最终奖励 R = (1-α) × R_outcome + α × R_outcome × R̂_rubric
```

其中：
- `R_outcome` ∈ {0, 1}：最终答案是否正确（二元）
- `R̂_rubric` ∈ [0, 1]：Agent 的推理过程满足了多少原子评估标准
- `α = 0.3`：平衡系数

关键设计：**rubric reward 只在答案正确时才生效**（乘以 R_outcome）。这防止了模型为了获取 rubric 分数而偏离正确答案的方向。

### 什么是 Rubric？

每个多跳问题在训练前被分解为若干**原子事实陈述**（rubric），使用隐藏实体占位符：

```
问题: "曾经完成了马纳萨卢的首次登顶的一名登山者来自哪个国家？"

Rubric 示例:
- "<E1> 是完成 <E2> 首次登顶的登山者之一"
- "<E2> 是一座海拔 8,163 米的山峰"
- "<E0> 是 <E1> 的国籍"
```

其中 `<E0>` 是最终答案，`<E1>` 到 `<E8>` 是中间实体。这些占位符确保 Agent 必须通过搜索找到每个实体，而不是直接猜。

### CaRR 三步评估流程

1. **隐藏实体识别**：从 Agent 的回答中提取每个实体的值（防止捷径）
2. **引文验证**：检查 Agent 引用的网页是否真的支持其声明（防止幻觉）
3. **证据连通性**：用 BFS 检查 rubric 之间的逻辑连接（防止 rubric 刷分）

### 各组件说明

| 组件 | 角色 | 来源 |
|------|------|------|
| **RayPPOTrainer** | 主训练循环：加载 batch → rollout → reward → advantage → 策略更新 | verl 框架自带 |
| **AgentLoopManager** | 编排多个 Agent 并行执行 rollout，管理推理服务器 | verl 框架自带 |
| **CaRRToolAgentLoop** | 自定义 Agent 循环，维护 CaRR 格式的 reward history | **需要实现** |
| **CaRRBrowserTool** | 封装 browser.search/open/find 三个工具，通过 HTTP 调用 CaRR Tool Server | **需要实现** |
| **CaRR Tool Server** | 外部服务，提供网页搜索（Serper API）、网页打开（Jina Reader）、页面内搜索 | CaRR 代码库自带 |
| **CaRR Reward Server** | 外部服务，执行三步 rubric 评估，使用 DeepSeek 作为 Judge LLM | CaRR 代码库自带 |
| **NaiveRewardManager** | verl 的 reward 管理器，调用 `compute_score` 函数 | verl 框架自带 |
| **C-GRPO Advantage** | 自定义 advantage estimator，融合 outcome 和 rubric reward | **需要实现** |

### 阶段 C: RL 训练（C-GRPO）

这是最复杂的部分。每个训练步骤的数据流如下：

```
RL 训练步 (一个 batch = 128 条问题，每条 rollout 16 次 = 2048 条轨迹)
    │
    │ ① 从 parquet 加载一个 batch 的 prompt
    ▼
┌────────────────────────────────────┐
│  AgentLoopManager.generate()       │
│  每条 prompt 独立跑一个 Agent:      │
│                                    │
│  [LLM 生成] ←→ [工具调用]           │
│      │              │              │
│      │    ┌─────────▼──────────┐   │
│      │    │ CaRRBrowserTool    │   │
│      │    │   → HTTP POST      │   │
│      │    │   → Tool Server    │   │
│      │    │   :7230            │   │
│      │    └────────────────────┘   │
│      │                             │
│      │  循环最多 30 轮              │
│      ▼                             │
│  最终输出:                          │
│  - token_ids (prompt + responses)  │
│  - response_mask (哪些是LLM生成的)  │
│  - response_logprobs (对数概率)     │
│  - extra_fields:                   │
│      messages (CaRR reward history)│
│      task_unfinished (是否被截断)   │
└──────────────┬─────────────────────┘
               │
               │ ② Reward 计算
               ▼
┌────────────────────────────────────┐
│  NaiveRewardManager                │
│  调用 carr_reward.compute_score()  │
│                                    │
│  构造 payload:                     │
│  - history (来自 extra_fields)     │
│  - label (ground_truth)            │
│  - rubrics, search_forbidden_strs  │
│                                    │
│     → HTTP POST                    │
│     → Reward Server :8888          │
│     → 返回 outcome_reward,         │
│       rubric_reward                │
│                                    │
│  写入 non_tensor_batch:            │
│  - score (= outcome_reward)        │
│  - outcome_reward                  │
│  - rubric_reward                   │
└──────────────┬─────────────────────┘
               │
               │ ③ Advantage 计算
               ▼
┌────────────────────────────────────┐
│  compute_advantage()               │
│  → C-GRPO estimator               │
│                                    │
│  对每个 group (同一问题的16条轨迹): │
│  1. norm_rubric = rubric / max     │
│  2. R = (1-α)·outcome              │
│       + α·outcome·norm_rubric      │
│  3. advantage = R - mean(R_group)  │
│  4. 标准化 advantage               │
└──────────────┬─────────────────────┘
               │
               │ ④ 策略更新
               ▼
┌────────────────────────────────────┐
│  Actor (Policy) Update             │
│  - 只对 response_mask=1 的 token   │
│    计算 PPO loss                   │
│  - mini-batch 梯度更新             │
│  - KL 散度控制（可选）             │
└────────────────────────────────────┘
```

## 1.6 verl 已完成的 2 处核心修改详解

### 修改 1: AlgoConfig 新增 `cgrpo_alpha`

**文件**: `verl/trainer/config/algorithm.py:615`

```python
class AlgoConfig(BaseConfig):
    gamma: float = 1.0
    lam: float = 1.0
    adv_estimator: str = "gae"
    norm_adv_by_std_in_grpo: bool = True
    use_kl_in_reward: bool = False
    kl_penalty: str = "kl"
    kl_ctrl: KLControlConfig = field(default_factory=KLControlConfig)
    # ... 其他字段 ...
    rollout_correction: Optional[RolloutCorrectionConfig] = None
    cgrpo_alpha: float = 0.3   # <--- 新增：C-GRPO rubric reward 混合比例
```

**为什么需要**：C-GRPO 算法需要一个 α 参数来控制 outcome reward 和 rubric reward 的混合比例。将其放在 `AlgoConfig` 中可以通过 Hydra 配置文件或命令行覆盖。

**注意**：修改 `AlgoConfig` 后，pre-commit hook `autogen-trainer-cfg` 会自动重新生成 `_generated_*.yaml` 文件。

### 修改 2: compute_advantage 条件传递 non_tensor_batch

**文件**: `verl/trainer/ppo/ray_trainer.py:215-220`

```python
def compute_advantage(data, config):
    # ... 前面的逻辑 ...

    else:
        # handle all other adv estimator type other than GAE and GRPO
        adv_estimator_fn = core_algos.get_adv_estimator_fn(adv_estimator)
        adv_kwargs = {
            "token_level_rewards": data.batch["token_level_rewards"],
            "response_mask": data.batch["response_mask"],
            "config": config,
        }
        # ... 其他可选参数 ...

        # ↓↓↓ 新增代码 ↓↓↓
        # Conditionally pass non_tensor_batch only to estimators that accept it
        _sig = inspect.signature(adv_estimator_fn)
        if "non_tensor_batch" in _sig.parameters or any(
            p.kind == inspect.Parameter.VAR_KEYWORD
            for p in _sig.parameters.values()
        ):
            adv_kwargs["non_tensor_batch"] = data.non_tensor_batch
        # ↑↑↑ 新增代码 ↑↑↑

        advantages, returns = adv_estimator_fn(**adv_kwargs)
```

**为什么需要**：
- C-GRPO 的 advantage estimator 需要访问 `non_tensor_batch` 中的 `outcome_reward` 和 `rubric_reward`
- 但现有的 estimator（如 `OPTIMAL_TOKEN_BASELINE`）的签名中没有 `non_tensor_batch` 参数
- 如果直接把 `non_tensor_batch` 传给它们，会因为未知参数报错

**解决方案**：使用 `inspect.signature()` 检查函数签名：
- 如果函数显式声明了 `non_tensor_batch` 参数 → 传递
- 如果函数接受 `**kwargs` → 传递
- 否则 → 不传递

这确保了向后兼容性：现有 estimator 不受影响，新的 C-GRPO estimator 能收到所需数据。

**数据流上下文**：
```
                                    non_tensor_batch 包含：
                                    ├── uid (样本唯一标识)
carr_reward.compute_score()         ├── score (= outcome_reward)
  返回 dict ──────────────────────> ├── outcome_reward (0 或 1)
                                    └── rubric_reward (0~1 之间的浮点数)
                                           │
                                           ▼
                              compute_advantage() 传给 C-GRPO estimator
```

## 1.7 关键概念速查

| 概念 | 说明 |
|------|------|
| **DataProto** | verl 的通用数据交换格式，包含 `batch`（张量）、`non_tensor_batch`（非张量，如字符串）、`meta_info`（元数据） |
| **response_mask** | 标记每个 token 是否由 LLM 生成（1=LLM 生成→参与梯度，0=工具/交互插入→只提供上下文） |
| **Delta-based Tokenization** | 多轮对话中，每轮只对新增部分做 tokenize，避免编码解码不一致 |
| **Sticky Session** | 同一个样本的多轮推理总是路由到同一台推理服务器，利用 KV cache |
| **Streaming Reward** | 每个样本 rollout 完成后立即计算 reward，不等整个 batch 完成 |
| **Registry Pattern** | verl 通过装饰器（如 `@register_adv_est("cgrpo")`）注册新组件，运行时动态查找 |
| **External Module** | 通过 `VERL_USE_EXTERNAL_MODULES` 环境变量加载外部 Python 模块，触发其中的注册装饰器 |
