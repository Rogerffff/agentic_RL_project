# Part 2: 数据预处理

本部分详细介绍 CaRR 原始数据的格式、verl 要求的目标格式，以及两者之间的转换逻辑。

---

## 2.1 两份原始数据文件

CaRR 提供了两份数据，分别用于 SFT 和 RL 阶段：

| 数据文件 | 用途 | 条数 | 大小 |
|---------|------|------|------|
| `CaRR/data/deepdive-sft-glm46-trajectory-1k.jsonl` | SFT 冷启动 | 832 条完整轨迹 | ~151 MB |
| `CaRR/data/deepdive-rl-2k-rubrics.jsonl` | RL 训练 | 2234 条问题+rubrics | ~8.8 MB |

**两份数据的本质区别**：
- **SFT 数据**包含完整的多轮对话轨迹（问题→搜索→打开网页→再搜索→...→最终回答），是"标准答案"
- **RL 数据**只包含问题和评估标准（rubrics），不包含轨迹——轨迹由模型自己生成

## 2.2 RL 数据格式详解

### 2.2.1 原始格式（CaRR JSONL）

每一行是一个 JSON 对象，包含以下字段：

```json
{
  "source": "deepsearch_with_browser",
  "input_messages": [
    {
      "role": "user",
      "content": "In the field of genome rearrangement, breakpoint graphs are a vital tool..."
    }
  ],
  "label": "Can a Breakpoint Graph Be Decomposed into None Other Than 2-Cycles?",
  "tools": [
    {
      "name": "browser.search",
      "description": "Search in browser",
      "parameters": {
        "type": "object",
        "properties": {
          "query": { "description": "Search query", "type": "string" },
          "num": { "description": "Number of results to return", "type": "integer", "default": 10 }
        },
        "required": ["query"]
      }
    },
    {
      "name": "browser.open",
      "description": "Open browser link",
      "parameters": {
        "type": "object",
        "properties": {
          "id": { "description": "ID or URL", "type": ["integer", "string"] }
        },
        "required": ["id"]
      }
    },
    {
      "name": "browser.find",
      "description": "Find pattern in the opened browser content",
      "parameters": {
        "type": "object",
        "properties": {
          "pattern": { "description": "Pattern to find", "type": "string" }
        },
        "required": ["pattern"]
      }
    }
  ],
  "metadata": {
    "remote_env_info": {
      "search_forbidden_strs": [
        "In the field of genome rearrangement, breakpoint graphs are a vital tool..."
      ],
      "rubrics": [
        "<E2> is a breakpoint graph, a tool used in the field of genome rearrangement <E1>.",
        "A basic structural question <E3> regarding <E2> asks if ...",
        "<E0> is the exact title of a paper.",
        "The paper with title <E0> directly investigates <E3>.",
        "The paper with title <E0> was published in ... <E4>.",
        "The paper with title <E0> was published in Volume 734, pages 38-45 <E5>.",
        "The paper with title <E0> was published circa the year <E6>, which is 2017."
      ]
    }
  }

```

**各字段含义**：

| 字段 | 说明 |
|------|------|
| `source` | 数据来源标识，固定为 `"deepsearch_with_browser"` |
| `input_messages` | 用户问题，只有一条 user message |
| `label` | 标准答案（简短的实体字符串），用于 outcome reward 判断 |
| `tools` | 三个浏览器工具的 schema 定义（CaRR 自定义格式） |
| `metadata.remote_env_info.search_forbidden_strs` | 防作弊字符串列表。通常包含原始问题文本，防止 Agent 直接搜索问题本身 |
| `metadata.remote_env_info.rubrics` | 原子评估标准列表（5-15 条），使用 `<E0>`-`<E8>` 占位符 |

### 2.2.2 目标格式（verl RL Parquet）

verl 的 RL 数据集（`RLHFDataset`）要求以下字段结构。我们来看 verl 是如何读取的：

**数据加载入口**：`verl/utils/dataset/rl_dataset.py:341-362`

```python
def __getitem__(self, item):
    row_dict: dict = self.dataframe[item]
    row_dict["raw_prompt"] = self._build_messages(row_dict)  # 把 prompt 字段处理为 messages 列表

    # 从 extra_info 中提取关键字段
    tools_kwargs = row_dict.get("extra_info", {}).get("tools_kwargs", {})
    interaction_kwargs = row_dict.get("extra_info", {}).get("interaction_kwargs", {})
    need_tools_kwargs = row_dict.get("extra_info", {}).get("need_tools_kwargs", self.need_tools_kwargs)

    row_dict["tools_kwargs"] = tools_kwargs     # ← 传递给工具的创建参数
    row_dict["interaction_kwargs"] = interaction_kwargs
    return row_dict
```

因此，我们需要输出的 Parquet 每一行包含：

```json
{
  "data_source": "carr_deepsearch",
  "agent_name": "carr_tool_agent",
  "prompt": [
    {"role": "user", "content": "In the field of genome rearrangement..."}
  ],
  "ability": "deepsearch",
  "reward_model": {
    "style": "rule",
    "ground_truth": "Can a Breakpoint Graph Be Decomposed into None Other Than 2-Cycles?"
  },
  "extra_info": {
    "split": "train",
    "index": 0,
    "rubrics": ["<E2> is a breakpoint graph...", "..."],
    "search_forbidden_strs": ["In the field of genome rearrangement..."],
    "rubric_reward_ratio": 0.3,
    "need_tools_kwargs": true,
    "tools_kwargs": {
      "browser.search": {
        "create_kwargs": {
          "search_forbidden_strs": ["..."]
        }
      },
      "browser.open": {
        "create_kwargs": {
          "search_forbidden_strs": ["..."]
        }
      },
      "browser.find": {
        "create_kwargs": {}
      }
    }
  }

```

**逐字段说明**：

| 字段 | 值 | 作用 | 被谁使用 |
|------|---|------|---------|
| `data_source` | `"carr_deepsearch"` | 数据源标识，reward 函数根据此判断调用哪个 compute_score | NaiveRewardManager |
| `agent_name` | `"carr_tool_agent"` | Agent Loop 类型名，从 registry 查找对应的 AgentLoop 类 | AgentLoopWorker |
| `prompt` | messages 列表 | 首版直接沿用原始 RL 数据的 `input_messages` | AgentLoop.run() |
| `ability` | `"deepsearch"` | 能力标签，用于分组统计 | 日志/评估 |
| `reward_model.style` | `"rule"` | 使用基于规则的自定义 reward 函数 | RewardManager |
| `reward_model.ground_truth` | 标准答案字符串 | 传递给 compute_score 的 ground_truth 参数 | carr_reward.py |
| `extra_info.rubrics` | rubric 列表 | 传递给 CaRR reward server 用于细粒度评估 | carr_reward.py |
| `extra_info.search_forbidden_strs` | 防作弊字符串 | 传递给 CaRR tool server 和 reward server | carr_browser_tool.py, carr_reward.py |
| `extra_info.rubric_reward_ratio` | `0.3` | 即 α，C-GRPO 混合比例 | carr_reward.py |
| `extra_info.need_tools_kwargs` | `true` | 告诉 verl 需要从 extra_info 中提取 tools_kwargs | RLHFDataset |
| `extra_info.tools_kwargs` | 工具创建参数 | 每个工具的 `create_kwargs`，传递给 `tool.create()` | AgentData → CaRRBrowserTool |

### 2.2.3 System Prompt 结论

这里要特别澄清：**`verl` 的 RL 数据格式不要求必须包含显式 `system` message**。

虽然很多示例数据使用 `[system, user]` 结构，但 `RLHFDataset` 实际读取时只是把 `prompt` 字段原样作为 messages 列表取出，再交给 `AgentLoop` 的 `apply_chat_template(messages, tools=...)` 处理。也就是说，只要 `prompt` 是合法的 chat messages 列表，就能工作。

对当前 CaRR 集成，**首版建议不额外添加 system prompt**，直接沿用原始 RL 数据中的 `input_messages`（即单条 `user` 消息）。原因如下：

1. **保持数据分布一致**：CaRR 原始 SFT / RL 数据都不带显式 `system`，首版尽量不要引入额外 prompt 偏移。
2. **原始 user 已包含输出格式要求**：问题文本本身已经要求模型按 `## Explanation with Citations / ## Exact Answer / ## References` 输出。
3. **工具协议由框架自动注入**：不要手写 `# Tools`、`<tools>`、`<tool_call>` 这些内容；在 `verl` 中，当 `tools` schema 传给 `apply_chat_template(...)` 时，chat template 会自动把工具调用协议拼进最终 prompt。

因此，RL 预处理阶段的推荐做法是：

- `prompt = input_messages`
- 不手写工具调用协议文本
- 不在 `system` 中重复写答案格式要求

如果后续做 ablation，想比较“有无 system prompt”的效果，可以额外尝试一个**极短**版本。但要注意：如果这样做，最好保持 SFT 与 RL 一致都加上。推荐仅使用：

```text
You are a research assistant. Use the available browser tools to gather evidence step by step when needed. Support claims with citations and follow the user's requested answer format.
```

### 2.2.4 tools_kwargs 的数据流

这是一个容易混淆的部分。让我们跟踪 `tools_kwargs` 如何从 Parquet 流到工具执行：

```
Parquet extra_info.tools_kwargs
    │
    │ RLHFDataset.__getitem__()
    │ row_dict["tools_kwargs"] = extra_info["tools_kwargs"]
    ▼
DataProto.non_tensor_batch["tools_kwargs"]
    │
    │ AgentLoopWorker._process_single_sample()
    │ tools_kwargs = non_tensor_batch["tools_kwargs"]
    ▼
AgentData(tools_kwargs=tools_kwargs)
    │
    │ CaRRBrowserTool.execute(agent_data=agent_data)
    │ create_kwargs = agent_data.tools_kwargs.get(self.name, {}).get("create_kwargs", {})
    │ search_forbidden_strs = create_kwargs.get("search_forbidden_strs", [])
    ▼
传递给 CaRR Tool Server 的 remote_env_info
```

**关键点**：`tools_kwargs` 是一个嵌套字典，key 是工具名（如 `"browser.search"`），value 中包含 `create_kwargs`。这种结构让每个工具都能接收自己特定的参数。

### 2.2.5 对比 GSM8K 参考示例

我们的 RL 预处理脚本参考 `examples/data_preprocess/gsm8k_tool_agent_loop.py`。对比关键差异：

| 方面 | GSM8K | CaRR DeepSearch |


### 2.3.2 格式差异：CaRR 格式 vs Qwen3 格式

CaRR 的数据格式（来自 GLM-4 生成）和 Qwen3 的 chat template 期望的格式有三处关键差异。如果不做转换，`tokenizer.apply_chat_template()` 会报错：

#### 差异 1: tool_calls 格式

**CaRR 格式**：
```json
{
  "tool_call_id": "tool-call-xxx",
  "name": "browser.search",
  "arguments": {"query": "...", "num": 10}    // ← dict 对象
}
```

**Qwen3 格式**（OpenAI function calling 标准）：
```json
{
  "type": "function",
  "id": "tool-call-xxx",
  "function": {
    "name": "browser.search",
    "arguments": "{\"query\": \"...\", \"num\": 10}"    // ← JSON 字符串
  }
}
```

差异要点：
- 多了 `"type": "function"` 字段
- `tool_call_id` → `id`
- `name` 和 `arguments` 包裹在 `function` 子对象中
- `arguments` 从 dict 变为 **JSON 字符串**

#### 差异 2: tool 消息的 content 格式

**CaRR 格式**：
```json
{
  "role": "tool",
  "content": [{"output": "search results text..."}]    // ← list of dict
}
```

**Qwen3 格式**：
```json
{
  "role": "tool",
  "content": "search results text..."    // ← 纯字符串
}
```

#### 差异 3: assistant.content 为 null

**CaRR 格式**：当 assistant 发起工具调用时，`content` 为 `null`（Python 的 `None`）

**Qwen3 格式**：需要 `content` 为空字符串 `""`

#### 差异 4: tools 定义格式

**CaRR 格式**（平铺）：
```json
{
  "name": "browser.search",
  "description": "Search in browser",
  "parameters": { ... }
}
```

**OpenAI function tools 格式**（包裹一层）：
```json
{
  "type": "function",
  "function": {
    "name": "browser.search",
    "description": "Search in browser",
    "parameters": { ... }
  }
}
```

### 2.3.3 转换函数

实施方案中提供了两个核心转换函数：

**消息转换** (`convert_message`)：

```python
import json

def convert_message(msg):
    """Convert CaRR message format to Qwen3 chat template format."""
    converted = {"role": msg["role"]}

    if msg["role"] == "assistant":
        # content: null → ""
        converted["content"] = msg.get("content") if msg.get("content") is not None else ""
        # 保留 reasoning_content（Qwen3 thinking mode 使用）
        if msg.get("reasoning_content"):
            converted["reasoning_content"] = msg["reasoning_content"]
        # 转换 tool_calls 格式
        if msg.get("tool_calls"):
            converted["tool_calls"] = [
                {
                    "type": "function",
                    "id": tc.get("tool_call_id", f"call_{i}"),
                    "function": {
                        "name": tc["name"],
                        "arguments": json.dumps(tc["arguments"], ensure_ascii=False)
                        if isinstance(tc.get("arguments"), dict)
                        else str(tc.get("arguments", "{}")),
                    },
                }
                for i, tc in enumerate(msg["tool_calls"])
            ]
    elif msg["role"] == "tool":
        # content: [{"output": "..."}] → "..."
        content = msg.get("content", "")
        if isinstance(content, list) and len(content) > 0 and isinstance(content[0], dict):
            converted["content"] = content[0].get("output", "")
        else:
            converted["content"] = str(content)
    else:
        converted["content"] = msg.get("content", "")

    return converted
```

**工具定义转换** (`convert_tools`)：

```python
def convert_tools(tools):
    """Convert CaRR tool schema to OpenAI format."""
    return [
        {
            "type": "function",
            "function": {
                "name": t["name"],
                "description": t.get("description", ""),
                "parameters": t.get("parameters", {}),
            },
        }
        for t in tools
    ]
```

### 2.3.4 目标格式（verl SFT Parquet）

verl 的 SFT 数据集（`MultiTurnSFTDataset`）读取逻辑在 `verl/utils/dataset/multiturn_sft_dataset.py:84-192`：


因此 SFT Parquet 每一行需要：

**关键字段说明**：

| 字段 | 说明 |
|------|------|
| `messages` | 转换后的完整多轮对话（Qwen3 格式） |
| `tools` | 转换后的工具定义（OpenAI function tools 格式） |
| `enable_thinking` | `true` — 启用 Qwen3 的 thinking mode（处理 `reasoning_content`） |

### 2.3.5 SFT 数据加载时的 tokenize 过程

理解 `MultiTurnSFTDataset.__getitem__()` 的行为很重要（`multiturn_sft_dataset.py:297-314`）：

```python
def __getitem__(self, item):
    messages = self._build_messages(row_dict)
    tools = self.tools[item] if self.tools is not None else None
    enable_thinking = self.enable_thinking[item] if ...

    # 逐条消息 tokenize
    for i, message in enumerate(messages):
        _input_ids, _loss_mask, _attention_mask, _inputs = self._process_single_message(
            index=i,
            message=message,
            full_message=messages,
            tools=tools if i == 0 else None,       # ← tools 只在第一条消息时传入
            enable_thinking=enable_thinking,
        )
```

注意 `tools=tools if i == 0 else None`：工具定义只在处理**第一条消息**时传给 `apply_chat_template`。这是因为 Qwen3 的 chat template 会把工具定义编码到系统消息中，后续消息不需要再传。

每条消息单独 tokenize 后，通过 `loss_mask` 控制哪些 token 参与训练：
- `assistant` 消息：`loss_mask = 1`（参与训练损失）
- `user`/`tool`/`system` 消息：`loss_mask = 0`（只提供上下文）

### 2.3.6 reasoning_content 的处理

SFT 数据中的 `reasoning_content` 是 GLM-4 的思考过程。在 Qwen3 中，当 `enable_thinking=True` 时，`apply_chat_template` 会自动把 `reasoning_content` 包裹在 `<think>...</think>` 标签中。

这意味着模型在 SFT 阶段就学会了"先思考再行动"的模式，RL 阶段模型也会延续这个习惯。

### 2.3.7 对比 GSM8K SFT 参考示例

`examples/data_preprocess/gsm8k_multiturn_sft.py` 非常简单——只有 user + assistant 两条消息，没有工具调用：

```python

    "messages": [
        {"role": "user", "content": question},
        {"role": "assistant", "content": answer_raw},

```

我们的 SFT 数据要复杂得多：88 条消息（多轮工具调用）、需要格式转换、有 thinking 内容。这也是为什么需要单独写预处理脚本。

## 2.4 预处理脚本设计要点

### 2.4.1 preprocess_carr_rl.py 核心逻辑

```python
            "search_forbidden_strs": forbidden,
            "rubric_reward_ratio": 0.3,
            "need_tools_kwargs": True,
            "tools_kwargs": tools_kwargs,
        },
    }

# 5. 切分 train/val（95%/5%，seed=42）
# 6. 保存为 Parquet
```

**注意事项**：
- 必须保证 `search_forbidden_strs` 非空（CaRR reward server 用 `[0]` 获取问题文本）
- 使用固定随机种子 `seed=42` 确保可复现
- 2234 条中取 5% 作为 val ≈ 112 条

### 2.4.2 preprocess_carr_sft.py 核心逻辑

```python
# 伪代码
valid_rows = []
for idx, line in enumerate(jsonl_lines):
    item = json.loads(line)

    # 1. 转换每条消息

    # 4. 收集有效样本

        "messages": converted_messages,

        "enable_thinking": True,

# 5. 切分 train/val（95%/5%，seed=42）
# 6. 保存为 Parquet
```

## 2.6 小结

数据预处理是最直接的一步，没有外部服务依赖。核心要点：

1. **RL 数据**：从原始 JSONL 提取字段，直接沿用 `input_messages` 作为 `prompt`，构造 `extra_info`（含 rubrics、tools_kwargs），保存 Parquet
2. **SFT 数据**：做三处格式转换（tool_calls 格式、tool content 格式、tools 定义格式），逐条 tokenize 验证，保存 Parquet
3. **验证**：RL 检查字段完整性，SFT 检查 `apply_chat_template` 能否通过
