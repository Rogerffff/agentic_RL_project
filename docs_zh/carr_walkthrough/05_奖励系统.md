# Part 5: 奖励系统

本部分详细讲解 CaRR 的三步 rubric reward 评估流程、Reward Server API、verl 中奖励数据的完整流动路径，以及我们需要实现的 `carr_reward.py`。

---

## 5.1 CaRR 奖励评估的核心思想

传统 Deep Search Agent 的奖励通常只有 outcome reward（答对/答错），这种二元信号非常稀疏——一个多步搜索轨迹可能包含 10+ 次工具调用，但最终只得到一个 0 或 1 的信号。

CaRR 的核心创新是引入 **rubric reward**（评分细则奖励），将一个复杂问题分解为多个原子性评估维度，并使用 **隐藏实体占位符** 防止答案泄露：

```
原始 rubric（带占位符）:
  C1. The response correctly identifies that <E1> is <E2>'s <E3>.
  C2. The response provides evidence that <E1> was born in <E4>.

隐藏实体映射（从模型回答中提取）:
  E0 = "final answer"  (特殊：始终是最终答案)
  E1 = "John Smith"
  E2 = "Alice Johnson"
  ...

填充后的 rubric:
  C1. The response correctly identifies that John Smith is Alice Johnson's father.
  C2. The response provides evidence that John Smith was born in London.
```

这样做的好处：
1. **训练时不泄露答案**：rubrics 中只有占位符 `<E1>`，模型必须自己搜索才能找到实体值
2. **细粒度信号**：每个 rubric 独立评判，提供 N 维奖励向量而非单一标量
3. **证据溯源**：通过引用检查确保模型不是"猜"出答案，而是基于证据推理

## 5.2 CaRR Reward Server 完整解析

### 5.2.1 服务器架构

**文件**: `CaRR/deepsearch_rm_with_rubrics/launch_server.py`

```
FastAPI Server (port 8888)
    │
    ├── GPTModel (LLM Judge)
    │   └── 使用 DeepSeek / OpenAI 兼容 API
    │       - model_name: "deepseek-chat"
    │       - 3 次重试
    │
    ├── /evaluate (POST 端点)
    │   ├── 输入验证
    │   ├── get_reward() → 综合奖励计算
    │   └── 返回: {reward, outcome_reward, rubric_reward, rubric_scores}
    │
    └── Prompt 模板 (./prompts/)
        ├── get_outcome_reward.txt    → 判断答案正确性
        ├── identify_entity.txt       → 识别隐藏实体
        └── judge_rubric.txt          → 评判 rubric 是否满足
```

### 5.2.2 `/evaluate` 端点请求/响应格式

**请求体**（`launch_server.py:459-512`）:

```json
{
  "history": [
    {"role": "user", "content": "..."},
    {"role": "assistant", "content": "...", "tool_calls": [...]},
    {"role": "tool", "content": [{"tool_call_id": "...", "output": "..."}]},
    {"role": "assistant", "content": "final answer with [1](url) citations"}
  ],
  "label": "correct answer string",
  "task_unfinished": false,
  "remote_env_info": {
    "search_forbidden_strs": ["original question text"],
    "rubrics": ["C1. ...", "C2. ...", ...],
    "rubric_reward_ratio": 0.3
  }
}
```

**响应体**:

```json
{
  "reward": 0.85,
  "outcome_reward": 1.0,
  "rubric_reward": 0.67,
  "rubric_scores": {
    "0": {"raw_rubric": "...", "filled_rubric": "...", "score": 1, ...},
    "1": {"raw_rubric": "...", "filled_rubric": "...", "score": 0, ...},
    ...
  }
}
```

### 5.2.3 关键验证逻辑

在执行评估前，reward server 有两个"快速返回 0"的条件（`launch_server.py:469-489`）:

```python
# 条件 1: 任务未完成
task_unfinished = data["task_unfinished"]
if task_unfinished:
    return {"reward": 0, "outcome_reward": 0, "rubric_reward": 0, "rubric_scores": {}}

# 条件 2: 最后一条消息不是 assistant
if history[-1]['role'] != 'assistant' or 'content' not in history[-1]:
    return {"reward": 0, "outcome_reward": 0, "rubric_reward": 0, "rubric_scores": {}}
```

这意味着：
- 如果 agent 被 `max_turns` 截断、未生成最终回答 → reward = 0
- 如果 `history` 最后一条是 tool 调用结果（role="tool"）→ reward = 0

> **对我们的影响**：`CaRRToolAgentLoop` 在 `run()` 结束时必须检查 `reward_history[-1].role` 是否为 `"assistant"`，如果不是则设置 `task_unfinished = True`。

## 5.3 三步 Rubric Reward 计算流程

### 5.3.1 总体流程

```
get_reward()  (launch_server.py:429-457)
    │
    ├── Step 0: 预处理 response（去掉 ## References 部分）
    │
    ├── if rubric_reward_ratio < 1:
    │   └── get_outcome_reward(response, question, answer)
    │       → outcome_reward ∈ {0, 1}
    │
    ├── if rubric_reward_ratio > 0:
    │   └── get_rubric_reward(response, question, answer, history, rubrics)
    │       → rubric_reward ∈ [0, 1]
    │
    └── final_reward = (1 - ratio) * outcome + ratio * rubric
```

默认 `rubric_reward_ratio = 0`（由 `remote_env_info` 传入），但我们在配置中会设置为 `0.3`。

### 5.3.2 Step 1: Outcome Reward（答案正确性判断）

`get_outcome_reward()` (`launch_server.py:47-75`):

1. 使用 `OUTCOME_REWARD_JUDGE_PROMPT` 模板，填入 question、correct_answer、response
2. 发送给 LLM Judge（DeepSeek），要求输出结构化结果
3. 用正则提取两个字段：
   ```python
   correctness_pattern = r"(?i)\*{0,2}correct\*{0,2}\s*:\s*(no|yes)"
   extracted_pattern = r"(?i)\*{0,2}extracted_final_answer\*{0,2}\s*:\s*(.+)"
   ```
4. 最多重试 3 次，返回 `accuracy = 1.0 if "yes" else 0`

> 这是 LLM-as-Judge 模式，比简单的字符串匹配更鲁棒。

### 5.3.3 Step 2: 隐藏实体识别（Hidden Entity Identification）

`get_rubric_reward()` 的第一阶段（`launch_server.py:299-326`）:

```
输入: rubrics（带 <E0>, <E1>, ... 占位符）, response（模型最终回答）
输出: all_entity_values = {"E0": "answer", "E1": "John", "E2": "Alice", ...}
```

流程：
1. 将所有 rubrics 拼接为编号文本 `C1. ... C2. ...`
2. 用 `extract_entity_ids()` 正则提取所有 `<Ex>` 占位符
3. 使用 `IDENTIFY_ENTITY_PROMPT` 让 LLM 从模型回答中识别每个实体的具体值
4. LLM 输出 JSON 块，用 `extract_json_block()` 解析
5. `check_identified_entities()` 验证：
   - 返回的是 dict
   - key 集合 == 所有 entity_ids
   - 值不为 None / "null" / "none"
6. 最多重试 5 次

### 5.3.4 Step 3: 引用验证与 Rubric 评判（Citation Judge）

`get_rubric_reward()` 的第二阶段（`launch_server.py:328-401`）:

**Step 3a — 填充 rubric**:

将识别出的实体值填入 rubric：

```python
# 对每个 rubric:
filled_rubric, all_filled, entity_values = fill_rubric_with_entity_value(rubric, all_entity_values)
# 如果某个实体未能识别（None），则该 rubric 直接得分 0
```

**Step 3b — 提取引用内容**:

`extract_citation_content()` (`launch_server.py:175-277`) 是一个关键函数。它从 history 中解析模型使用过的工具调用，构建 URL→内容的映射：

```
模型最终回答中的引用格式: [1](https://example.com)
                                ↓
从 history 中提取 tool_calls:
  assistant: tool_calls = [{tool_call_id: "tc_0", name: "browser.search", ...}]
  tool:      content = [{tool_call_id: "tc_0", output: "search results..."}]
                                ↓
构建 url_contents 映射:
  url → {search_results: [...], open_results: [...], find_results: [...]}
                                ↓
只保留被引用的 URL 的内容（最多 20 个）
```

核心逻辑：
1. 用正则 `\[\d+\]\((?:view-source:)?(https?://.+?)\)[\],]` 从 response 中提取所有被引用的 URL
2. 遍历 history，建立 `tool_call_id → tool_call` 的映射
3. 对每个 tool 输出：
   - `browser.search` → 解析搜索结果，提取 URL/title/snippet
   - `browser.open` → 解析页面内容，记录当前 URL
   - `browser.find` → 关联到当前 URL 的 find 结果
4. 将被引用 URL 的所有相关内容拼成 `citation_content` 字符串

**Step 3c — LLM 评判**:

使用 `JUDGE_RUBRIC_PROMPT` 让 LLM 判断每条填充后的 rubric 是否被 citation_content 所支持：

```python
judge_rubric_prompt = JUDGE_RUBRIC_PROMPT.format(
    context=citation_content,      # 引用来源的实际内容
    statements=rubric_text.strip() # 填充后的 rubric 列表
)
# LLM 返回: {"S1": true, "S2": false, ...}
```

### 5.3.5 Step 4: BFS 连通性检查（Evidence Connectivity）

`get_rubric_reward()` 的第三阶段（`launch_server.py:402-427`）:

这是 CaRR 的一个巧妙设计。即使某个 rubric 通过了引用验证，还需要检查它是否与最终答案（E0）连通：

```
实体关系图:
  E0 (final answer) ──rubric_1──> E1
  E1 ──rubric_2──> E2
  E3 ──rubric_3──> E4  (孤立，与 E0 不连通)

BFS from E0:
  - 从 E0 出发
  - 找到包含 E0 的已支持 rubric → rubric_1 可达
  - rubric_1 涉及 E1 → E1 可达
  - 找到包含 E1 的已支持 rubric → rubric_2 可达
  - ...
  - rubric_3 不可达（E3/E4 与 E0 不连通）→ score = 0
```

```python
# 构建 entity → rubrics 映射（仅已支持的 rubric）
entity2rubrics = defaultdict(set)
for idx, rubric in rubric_scores.items():
    if rubric["is_supported"] == 1:
        for entity in rubric["entity_values"]:
            entity2rubrics[entity].add(idx)

# BFS 从 E0 出发
arrivable_entites = {"E0"}
queue = deque(["E0"])
while queue:
    entity = queue.popleft()
    for idx in entity2rubrics[entity]:
        arrivable_rubrics.add(idx)
        for other_entity in rubric_scores[idx]["entity_values"]:
            if other_entity not in arrivable_entites:
                arrivable_entites.add(other_entity)
                queue.append(other_entity)

# 最终分数 = 可达且已支持的 rubric 数 / 总 rubric 数
rubric_reward = total_score / len(rubric_scores)
```

这保证了：孤立的、与最终答案无关的推理链不会获得奖励。

### 5.3.6 最终奖励合成

`get_reward()` (`launch_server.py:429-457`):

```python
final_reward = (1 - rubric_reward_ratio) * outcome_reward + rubric_reward_ratio * rubric_reward
```

当 `rubric_reward_ratio = 0.3` 时（CaRR 默认配置）：

| 场景 | outcome | rubric | final |
|------|---------|--------|-------|
| 答对 + 推理完整 | 1.0 | 0.8 | 0.7 + 0.24 = 0.94 |
| 答对 + 推理不完整 | 1.0 | 0.3 | 0.7 + 0.09 = 0.79 |
| 答错 + 推理完整 | 0.0 | 0.8 | 0.0 + 0.24 = 0.24 |
| 答错 + 推理不完整 | 0.0 | 0.0 | 0.0 |

## 5.4 verl 中奖励数据的完整流动路径

理解 `carr_reward.py` 如何被调用，需要追踪数据从 Agent Loop 到 Reward 的完整链路。

### 5.4.1 路径总览

```
CaRRToolAgentLoop.run()
    │
    │  extra_fields["messages"] = reward_history      ← AgentLoop 输出
    │  extra_fields["task_unfinished"] = bool
    │
    ▼
AgentLoopWorker._compute_score()              (agent_loop.py:690-718)
    │
    │  non_tensor_batch["tool_extra_fields"] = output.extra_fields
    │  → 发送给 reward_loop_worker
    │
    ▼
NaiveRewardManager.run_single()               (naive.py:34-99)
    │
    │  extra_info = data_item.non_tensor_batch["extra_info"]     ← 来自 parquet
    │  tool_extra_fields = data_item.non_tensor_batch["tool_extra_fields"]  ← 来自 AgentLoop
    │  extra_info.update(tool_extra_fields.items())              ← 合并！
    │
    │  此时 extra_info 包含:
    │    - "messages": reward_history          (来自 AgentLoop)
    │    - "task_unfinished": bool             (来自 AgentLoop)
    │    - "rubrics": [...]                    (来自 parquet extra_info)
    │    - "search_forbidden_strs": [...]      (来自 parquet extra_info)
    │
    ▼
carr_reward.compute_score(
    data_source="carr_deepsearch",
    solution_str=response_str,        ← tokenizer.decode(response_ids)
    ground_truth=ground_truth,        ← parquet 中的 reward_model.ground_truth
    extra_info=extra_info,            ← 合并后的完整 dict
)
    │
    ▼
HTTP POST → CaRR Reward Server /evaluate
    │
    ▼
返回 {"score": float, "outcome_reward": float, "rubric_reward": float}
```

### 5.4.2 Streaming Reward 路径的前提条件

CaRR 使用 **streaming reward** 路径，即在 agent loop 运行过程中异步计算奖励。这需要满足以下条件：

**条件来源** (`ray_trainer.py:832-844`):

```python
enable_agent_reward_loop = not self.use_rm or self.config.reward.reward_model.enable_resource_pool
```

当 `reward.reward_model.enable: false`（我们的配置），`self.use_rm = False`，所以 `enable_agent_reward_loop = True`。

此时：

```python
reward_loop_worker_handles = self.reward_loop_manager.reward_loop_workers  # 非 None
self.async_rollout_manager = AgentLoopManager(
    ...,
    reward_loop_worker_handles=reward_loop_worker_handles,  # 传入
)
```

在 `AgentLoopWorker._compute_score()` 中（`agent_loop.py:690-718`）:

```python
enable_async_reward = self.reward_loop_worker_handles is not None  # True

if output.reward_score is None and enable_async_reward:
    non_tensor_batch = {
        **{k: np.array([v]) for k, v in kwargs.items()},
        "__num_turns__": np.array([output.num_turns]),
        "tool_extra_fields": np.array([output.extra_fields], dtype=object),  # ← 关键！
    }
    # ... 发送给 reward_loop_worker
```

这里 `output.extra_fields` 就是 AgentLoop 中设置的 dict，包含 `messages` 和 `task_unfinished`。

**配置要点**:

```yaml
reward:
  reward_model:
    enable: false          # 不使用 RM 模型打分
    enable_resource_pool: false
  custom_reward_function:
    path: examples/carr_deepsearch/reward/carr_reward.py
    name: compute_score
  reward_manager:
    name: naive            # 使用 NaiveRewardManager
    source: register
```

### 5.4.3 extra_info 的两个来源

一个容易混淆的点是 `extra_info` 有两个来源，在 `NaiveRewardManager.run_single()` 中合并（`naive.py:44-47`）:

```python
# 来源 1: parquet 中的 extra_info（通过 RLHFDataset → non_tensor_batch 传入）
extra_info = data_item.non_tensor_batch.get("extra_info", {})
# 包含: rubrics, search_forbidden_strs 等静态数据

# 来源 2: AgentLoop 的 extra_fields（通过 tool_extra_fields 传入）
tool_extra_fields = data_item.non_tensor_batch.get("tool_extra_fields", None)
if tool_extra_fields is not None:
    extra_info.update(tool_extra_fields.items())
# 添加: messages, task_unfinished 等动态数据
```

这就是为什么我们在 Part 2（数据预处理）中将 `rubrics` 和 `search_forbidden_strs` 放入 parquet 的 `extra_info` 列，而在 Part 4（AgentLoop）中将 `messages` 和 `task_unfinished` 放入 `extra_fields` 的原因——两者最终在 NaiveRewardManager 中汇合。

## 5.5 carr_reward.py 实现详解

### 5.5.1 完整伪代码

**文件**: `examples/carr_deepsearch/reward/carr_reward.py`

```python
import logging
import os
import aiohttp

logger = logging.getLogger(__name__)

REWARD_SERVER_URL = os.environ.get("CARR_REWARD_SERVER_URL", "http://localhost:8888")
REWARD_TIMEOUT = int(os.environ.get("CARR_REWARD_TIMEOUT", "650"))


async def compute_score(data_source, solution_str, ground_truth, extra_info=None, **kwargs):
    """异步调用 CaRR Reward Server 的 /evaluate 端点。

    签名必须匹配 verl 的 compute_score 协议:
    - data_source: str — parquet 中的 data_source 列
    - solution_str: str — tokenizer.decode(response_ids)
    - ground_truth: str — parquet 中的 reward_model.ground_truth
    - extra_info: dict — 合并后的信息（来自 parquet + AgentLoop）
    - **kwargs: 可能包含 reward_router_address 等

    返回: dict with "score" key → 进入 rm_scores
    """
    extra_info = extra_info or {}

    # 从 extra_info 提取各字段
    messages = extra_info.get("messages", [])           # AgentLoop 的 reward_history
    rubrics = extra_info.get("rubrics", [])             # parquet extra_info
    search_forbidden_strs = extra_info.get("search_forbidden_strs", [])  # parquet extra_info
    rubric_reward_ratio = extra_info.get("rubric_reward_ratio", 0.3)
    task_unfinished = extra_info.get("task_unfinished", False)

    # Fallback: 如果 AgentLoop 没有传入 messages，构造最简 history
    if not messages:
        messages = [
            {"role": "user", "content": search_forbidden_strs[0] if search_forbidden_strs else "Question"},
            {"role": "assistant", "content": solution_str},
        ]

    # 安全检查: 最后一条不是 assistant 则标记未完成
    if messages and messages[-1].get("role") != "assistant":
        task_unfinished = True

    payload = {
        "history": messages,
        "label": ground_truth,
        "task_unfinished": task_unfinished,
        "remote_env_info": {
            "search_forbidden_strs": search_forbidden_strs,
            "rubrics": rubrics,
            "rubric_reward_ratio": rubric_reward_ratio,
        },
    }

    try:
        async with aiohttp.ClientSession() as session:
            async with session.post(
                f"{REWARD_SERVER_URL}/evaluate",
                json=payload,
                timeout=aiohttp.ClientTimeout(total=REWARD_TIMEOUT),
            ) as resp:
                if resp.status != 200:
                    logger.error("Reward server status=%s", resp.status)
                    return {"score": 0.0, "outcome_reward": 0.0, "rubric_reward": 0.0}
                result = await resp.json()
    except Exception as e:
        logger.error("Reward server call failed: %s", e)
        return {"score": 0.0, "outcome_reward": 0.0, "rubric_reward": 0.0}

    outcome_reward = float(result.get("outcome_reward", 0.0))
    rubric_reward = float(result.get("rubric_reward", 0.0))

    # score 作为 rm_scores 的值
    # C-GRPO 的 outcome/rubric 融合在 advantage 阶段完成（见 Part 6）
    return {
        "score": outcome_reward,
        "outcome_reward": outcome_reward,
        "rubric_reward": rubric_reward,
    }
```

### 5.5.2 关键设计决策

**1. 为什么 `score` 返回 `outcome_reward` 而非 `final_reward`？**

在 CaRR 原始实现中，reward server 在服务端计算：

```python
final_reward = (1 - ratio) * outcome + ratio * rubric
```

但我们在 verl 中采用 C-GRPO 算法，在 advantage 估计阶段进行更精细的融合（见 Part 6）：

```python
R_i = (1 - α) * R_outcome + α * R_outcome * R̂_rubric
```

这里 `R̂_rubric` 是 group-wise normalized 的 rubric reward。这种归一化必须在看到同一 prompt 的多个 rollout 后才能进行，因此不能在 reward server 端完成。

所以我们的策略是：
- `score` → `outcome_reward`，成为 `rm_scores`（用于 PPO/GRPO 基础 advantage）
- `rubric_reward` → 通过 `reward_extra_info` 传入 `non_tensor_batch`，在 advantage 阶段使用

**2. 为什么 `async def`？**

`compute_score` 是 async 函数，因为：
- 需要异步 HTTP 调用 reward server
- verl 的 `NaiveRewardManager` 检测到 `inspect.iscoroutinefunction(compute_score)` 为 True 时，会直接 `await` 而不是 `run_in_executor`
- 这避免了线程池开销，允许大量并发请求

**3. 超时为什么是 650 秒？**

CaRR reward server 内部有 `timeout=600` 的 `asyncio.wait_for`（`launch_server.py:492`）。客户端超时必须大于服务端超时，否则客户端先断开连接，导致结果丢失但服务端还在计算。650 > 600 提供了 50 秒缓冲。

**4. Fallback 逻辑**

当 `messages` 为空时（例如 AgentLoop 异常退出），构造最简 history：

```python
messages = [
    {"role": "user", "content": search_forbidden_strs[0]},
    {"role": "assistant", "content": solution_str},
]
```

这样 reward server 至少能做 outcome reward 评估，而非直接报错。

### 5.5.3 返回值如何进入 non_tensor_batch

回看 `NaiveRewardManager.run_single()` 的返回值处理（`naive.py:86-99`）:

```python
result = await self.compute_score(...)   # 我们的 carr_reward.compute_score

if isinstance(result, dict):             # 是 dict
    score = result["score"]              # → 成为 rm_scores
    for key, value in result.items():
        reward_extra_info[key] = value   # "outcome_reward", "rubric_reward" 都进入

return {"reward_score": reward, "reward_extra_info": reward_extra_info}
```

然后在 `AgentLoopWorker._compute_score()` 中（`agent_loop.py:717-718`）:

```python
output.reward_score = result["reward_score"]
output.extra_fields["reward_extra_info"] = result["reward_extra_info"]
```

最后在 `_postprocess()` 中（`agent_loop.py:768-771`）:

```python
reward_extra_infos = [input.extra_fields.get("reward_extra_info", {}) for input in inputs]
reward_extra_keys = list(reward_extra_infos[0].keys())
for key in reward_extra_keys:
    non_tensor_batch[key] = np.array([info[key] for info in reward_extra_infos])
```

结果：`non_tensor_batch["outcome_reward"]` 和 `non_tensor_batch["rubric_reward"]` 都可用于 C-GRPO advantage estimator。

## 5.8 与 CaRR 原始实现的差异

| 对比项 | CaRR 原始 | 我们的实现 |
|--------|-----------|-----------|
| reward 融合 | 在 server 端: `(1-r)*outcome + r*rubric` | 在 verl advantage 阶段（C-GRPO，见 Part 6）|
| `score` 含义 | = final_reward | = outcome_reward（rubric 另存） |
| history 格式 | GLM-4 格式 | Qwen3 格式（需在 AgentLoop 中构建 CaRR 兼容格式）|
| tool_call 格式 | `arguments` 是 dict | `arguments` 是 JSON string |
| 调用方式 | 直接 Python 调用 | HTTP 微服务（agent loop 和 reward server 分离）|

> **重要注意**：`reward_history` 中的消息格式必须与 CaRR reward server 期望的格式严格匹配。特别是 `extract_citation_content()` 函数（`launch_server.py:175-277`）会解析 `tool_calls` 中的 `tool_call_id`、`name`、`arguments`（JSON string）以及 tool role 消息中的 `content[].tool_call_id` 和 `content[].output`。如果格式不匹配，引用提取将失败，rubric reward 将为 0。
