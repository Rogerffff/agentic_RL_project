## 3.1 verl 工具系统架构

### 3.1.1 BaseTool 生命周期

verl 的工具系统定义在 `verl/tools/base_tool.py`。每个工具必须实现四个生命周期方法：

```
create(instance_id, **kwargs)       ← 创建工具实例（开始一次轨迹）
    │
    ├── execute(instance_id, parameters, **kwargs)   ← 执行工具（可调用多次）
    ├── execute(instance_id, parameters, **kwargs)
    ├── ...
    │
    ├── calc_reward(instance_id, **kwargs)   ← 计算基于工具状态的奖励（可选）
    │
    └── release(instance_id, **kwargs)       ← 释放资源
```

**`BaseTool` 源码**（`verl/tools/base_tool.py:24-93`）：

```python
class BaseTool:
    def __init__(self, config: dict, tool_schema: OpenAIFunctionToolSchema):
        self.config = config
        self.tool_schema = tool_schema or self.get_openai_tool_schema()
        self.name = self.tool_schema.function.name  # ← 工具名，如 "browser.search"

    async def create(self, instance_id=None, **kwargs) -> tuple[str, ToolResponse]:
        """创建工具实例，返回 (instance_id, 创建时的响应)"""
        if instance_id is None:
            return str(uuid4()), ToolResponse()
        return instance_id, ToolResponse()

    async def execute(self, instance_id, parameters, **kwargs) -> tuple[ToolResponse, float, dict]:
        """执行工具，返回 (响应, 步级奖励, 指标)"""
        return ToolResponse(text="Updated the tool state."), 0.0, {}

    async def calc_reward(self, instance_id, **kwargs) -> float:
        """计算工具级别奖励"""
        return 0.0

    async def release(self, instance_id, **kwargs) -> None:
        """释放工具实例资源"""
        pass
```

**关键数据结构**——`ToolResponse`（`verl/tools/schemas.py:94-123`）：

```python
class ToolResponse(BaseModel):
    text: str | None = None      # 文本输出（最常用）
    image: list[Any] | None = None   # 图片列表（多模态）
    video: list[Any] | None = None   # 视频列表（多模态）
```

我们的浏览器工具只返回文本，所以只用 `ToolResponse(text="...")`。

### 3.1.2 GSM8K 工具参考示例

`verl/tools/gsm8k_tool.py` 是一个很好的参考：

```python
class Gsm8kTool(BaseTool):
    def __init__(self, config, tool_schema):
        super().__init__(config, tool_schema)
        self._instance_dict = {}   # instance_id → 状态字典

    async def create(self, instance_id=None, ground_truth=None, **kwargs):
        if instance_id is None:
            instance_id = str(uuid4())
        # ← 注意: ground_truth 从 create_kwargs 获取
        if ground_truth is None:
            ground_truth = kwargs.get("create_kwargs", {}).get("ground_truth", None)
        self._instance_dict[instance_id] = {
            "response": "",
            "ground_truth": ground_truth,
            "reward": 0.0,
        }
        return instance_id, ToolResponse()

    async def execute(self, instance_id, parameters, **kwargs):
        answer = parameters.get("answer", "")
        # ... 计算奖励 ...
        return ToolResponse(text=f"Current parsed {answer=} {reward=}"), tool_reward, {}

    async def release(self, instance_id, **kwargs):
        del self._instance_dict[instance_id]   # ← 清理状态
```

**重要观察**：GSM8K 工具是**无状态服务**——每次 `create` 创建一个独立的实例状态，`release` 清理该状态。工具本身不需要管理"会话"概念。

### 3.1.3 ToolAgentLoop 如何调用工具

这是理解我们设计的关键。看 `tool_agent_loop.py:419-445` 的 `_call_tool` 方法：

```python
async def _call_tool(self, tool_call, tools_kwargs, agent_data):
    tool, instance_id = None, None
    try:
        tool_name = tool_call.name
        tool_args = json.loads(tool_call.arguments)
        tool = self.tools[tool_name]
        kwargs = tools_kwargs.get(tool_name, {})

        # ① create —— 每次工具调用都创建一个新实例
        instance_id, _ = await tool.create(create_kwargs=kwargs.get("create_kwargs", {}))

        # ② execute —— 执行工具
        tool_execution_response, tool_reward, res = await tool.execute(
            instance_id, tool_args, agent_data=agent_data  # ← agent_data 作为 kwarg 传入
        )
    except Exception as e:
        return ToolResponse(text=f"Error when executing tool: {e}"), 0.0, {}
    finally:
        if tool and instance_id:
            # ③ release —— 立即释放
            await tool.release(instance_id)

    return ...
```

**关键发现：每次工具调用都是完整的 create → execute → release 循环！**

这意味着：
- `instance_id` 每次调用都不同（`uuid4()` 生成）
- 不能靠 `instance_id` 维持跨轮次的会话状态
- 但 `agent_data` 始终是同一个对象（`execute` 通过 `kwargs` 接收）

### 3.1.4 工具注册（YAML 配置 + 动态加载）

`verl/tools/utils/tool_registry.py:82-142` 的 `initialize_tools_from_config` 函数从 YAML 加载工具：

```python
def initialize_tools_from_config(tools_config_file):
    tools_config = OmegaConf.load(tools_config_file)
    tool_list = []

    for tool_config in tools_config.tools:
        cls_name = tool_config.class_name          # "examples.carr_deepsearch.tools.carr_browser_tool.CaRRBrowserTool"
        tool_type = ToolType(tool_config.config.type)  # "native"
        tool_cls = get_tool_class(cls_name)        # 动态导入类

        # NATIVE 类型直接实例化
        tool_schema = OpenAIFunctionToolSchema.model_validate(tool_config.tool_schema)
        tool = tool_cls(
            config=OmegaConf.to_container(tool_config.config, resolve=True),
            tool_schema=tool_schema,
        )
        tool_list.append(tool)
    return tool_list
```

`get_tool_class` 通过 `importlib` 动态导入：

```python
def get_tool_class(cls_name):
    module_name, class_name = cls_name.rsplit(".", 1)  # "examples.carr_deepsearch.tools.carr_browser_tool", "CaRRBrowserTool"
    spec = importlib.util.find_spec(module_name)
    module = importlib.util.module_from_spec(spec)
    spec.loader.exec_module(module)
    tool_cls = getattr(module, class_name)
    return tool_cls
```

这意味着我们的工具类只需要放在正确的 Python 模块路径下，在 YAML 中用全限定名引用即可。

## 3.2 CaRR Tool Server API 详解

### 3.2.1 服务架构

CaRR Tool Server 是一个独立的 Quart（异步 Flask）HTTP 服务，运行在 `localhost:7230`。

**源码**：`CaRR/tool_server/launch_server.py`

核心结构非常简单——只有一个 POST 端点 `/`，所有操作通过 `name` 字段区分：

```python
app = Quart(__name__)
session2sandbox = defaultdict(dict)    # session_id → sandbox 状态字典

@app.route("/", methods=["POST"])
async def call_tool(tool_call):
    session_id = tool_call["session_id"]
    func_name = tool_call["name"]
    arguments = tool_call["arguments"]
    remote_env_info = tool_call["remote_env_info"]
    sandbox = session2sandbox[session_id]    # 获取该 session 的沙箱
    ...
```

### 3.2.2 请求格式

所有请求都是 POST JSON，格式统一：

```json
{
  "session_id": "unique_session_identifier",
  "name": "start_session | close_session | browser.search | browser.open | browser.find",
  "arguments": { ... },
  "remote_env_info": {
    "search_forbidden_strs": ["..."]
  }
}
```

### 3.2.3 五个操作

#### `start_session`
```python
if func_name == "start_session":
    result = "Sucess start session"
```
创建一个空的 sandbox 字典。因为 `session2sandbox` 是 `defaultdict(dict)`，访问 `session2sandbox[session_id]` 时自动创建。

#### `close_session`
```python
elif func_name == "close_session":
    if session_id in session2sandbox:
        del session2sandbox[session_id]    # 释放内存
    result = f"Sucessully close session {session_id}"
```
清理 sandbox 状态，释放内存。**如果不调用 close_session，sandbox 会一直占用内存。**

#### `browser.search(query, num)`
```python
elif func_name == "browser.search":
    query = arguments.get("query", "")
    num = arguments.get("num", 10)
    forbidden_strs = remote_env_info.get("search_forbidden_strs", [])
    result, idx2url = await search(query, num=num, forbidden_strs=forbidden_strs, ...)
    sandbox['idx2url'] = idx2url     # ← 存储搜索结果 ID 到 URL 的映射
```

调用 Serper API 进行 Google 搜索，返回格式：
```
[0] Title: ...
[0] URL Source: https://...
[0] Description: ...

[1] Title: ...
[1] URL Source: https://...
[1] Description: ...
```

**反作弊**：`forbidden_strs` 用于过滤包含原始问题文本的搜索结果（防止 Agent 直接搜索问题）。

关键：`sandbox['idx2url']` 保存了 `{0: "https://...", 1: "https://...", ...}` 映射，后续 `browser.open` 靠这个找到 URL。

#### `browser.open(id)`
```python
elif func_name == "browser.open":
    idx2url = sandbox.get("idx2url", {})     # ← 依赖之前 search 的结果
    idx = arguments.get("id", None)
    url = idx2url.get(idx) if isinstance(idx, int) else idx    # int → 查映射, str → 直接URL
    result = await parse_url(url=url, ...)
    sandbox["cur_web_content"] = result      # ← 存储网页内容供 find 使用
    result = result[:10000]                  # 截断到 10000 字符
```

调用 Jina Reader API 解析网页，返回标题+内容。

关键：`sandbox["cur_web_content"]` 保存了当前打开的网页内容，后续 `browser.find` 在这个内容中搜索。

#### `browser.find(pattern)`
```python
elif func_name == "browser.find":
    pattern = arguments.get("pattern", "")
    cur_web_content = sandbox.get("cur_web_content", "")    # ← 依赖之前 open 的结果
    find_results = find(pattern=pattern, parse_content=cur_web_content, ...)
    result = "\n\n".join([f"[Match {i+1}]\n{ctx}" for i, ctx in enumerate(find_results)])
```

在当前打开的网页中搜索模式，支持精确匹配和模糊匹配。

### 3.2.4 Session 的关键作用

从上面可以看出，**三个工具通过 sandbox 共享状态**：

```
browser.search → sandbox["idx2url"]       → browser.open 读取
browser.open   → sandbox["cur_web_content"] → browser.find 读取
```

如果每次工具调用使用不同的 session_id，这些状态就丢失了：
- `browser.open(id=0)` 找不到 `idx2url`，报错
- `browser.find(pattern="breakpoint")` 找不到 `cur_web_content`，返回空

**这就是为什么同一个 Agent 轨迹的所有工具调用必须使用同一个 session_id。**

## 3.3 设计挑战：工具生命周期 vs Session 生命周期

现在我们面对一个核心矛盾：

| | verl ToolAgentLoop | CaRR Tool Server |
|---|---|---|
| **粒度** | 每次工具调用 create → execute → release | 每个 session（一条完整轨迹） |
| **状态** | instance_id 每次都是新的 uuid | session_id 必须跨轮次保持一致 |
| **生命周期** | 调用级（毫秒） | 轨迹级（可能几分钟） |

**如果直接在 `create` 中 `start_session`，在 `release` 中 `close_session`**：
- 第一次 `browser.search`：start_session → search → close_session ✓
- 第二次 `browser.open`：start_session(新session) → open → idx2url 丢失 ✗

**解决方案**：把 session 管理从工具的 create/release 中抽出来，改为**请求级（request 级）管理**：

```
一条轨迹开始（AgentLoop.run 入口）
    │
    ├── 第1次工具调用: create → execute(确保session已启动) → release(不关session)
    ├── 第2次工具调用: create → execute(session已存在,复用) → release(不关session)
    ├── 第3次工具调用: create → execute(session已存在,复用) → release(不关session)
    │   ...
    │
一条轨迹结束（AgentLoop.run finally）
    └── close_session（统一关闭）
```

## 3.4 CaRRSessionManager 实现

这是一个**单例模式**的会话管理器，维护所有活跃 session 的状态：

```python
class CaRRSessionManager:
    _instance = None

    def __init__(self):
        self._started_sessions = set()                      # 已启动的 session_id 集合
        self._session_data: dict[str, dict[str, Any]] = {}  # session_id → 元数据

    @classmethod
    def get_instance(cls):
        if cls._instance is None:
            cls._instance = cls()
        return cls._instance
```

### 3.4.1 核心方法

**`ensure_started`**——懒启动 session：

```python
async def ensure_started(self, session_id: str, tool_server_url: str, **kwargs):
    if session_id not in self._started_sessions:
        await self.call_server(tool_server_url, session_id, "start_session", {}, {})
        self._started_sessions.add(session_id)
        self._session_data[session_id] = kwargs
```

这个方法会在第一次工具调用时自动创建 session。如果 session 已存在（第 2 次、第 3 次调用），直接跳过。

**`close`**——关闭 session：

```python
async def close(self, session_id: str, tool_server_url: str):
    if session_id in self._started_sessions:
        await self.call_server(tool_server_url, session_id, "close_session", {}, {})
        self._started_sessions.discard(session_id)
        self._session_data.pop(session_id, None)
```

只在 `AgentLoop.run()` 的 `finally` 中调用一次。

**`call_server`**——通用 HTTP 调用：

```python
async def call_server(self, url, session_id, name, arguments, remote_env_info):
    payload = {
        "session_id": session_id,
        "name": name,
        "arguments": arguments,
        "remote_env_info": remote_env_info,
    }
    async with aiohttp.ClientSession() as session:
        async with session.post(url, json=payload, timeout=aiohttp.ClientTimeout(total=60)) as resp:
            if resp.status != 200:
                return f"Tool server HTTP {resp.status}"
            result = await resp.json()
            return result.get("output", "")
```

### 3.4.2 session_id 的选择

用什么作为 session_id？答案是 `agent_data.request_id`：

- `request_id` 在 `AgentLoopWorker` 为每个样本生成，整条轨迹内保持不变
- 不同样本有不同的 `request_id`
- 它也用于 `AsyncLLMServerManager` 的 sticky session（同一 request_id 路由到同一推理服务器）

```python
# 在 CaRRBrowserTool.execute 中：
session_id = agent_data.request_id
```

### 3.4.3 为什么用单例模式？

因为同一个 YAML 配置创建了**3 个 CaRRBrowserTool 实例**（search、open、find），但它们需要共享同一个 session 管理器。单例确保：

```python
# browser.search 的工具实例
self.session_manager = CaRRSessionManager.get_instance()  # → 同一个对象

# browser.open 的工具实例
self.session_manager = CaRRSessionManager.get_instance()  # → 同一个对象

# browser.find 的工具实例
self.session_manager = CaRRSessionManager.get_instance()  # → 同一个对象
```

## 3.5 CaRRBrowserTool 实现

一个统一的工具类处理所有三个浏览器操作，通过 `self.name`（从 YAML 的 `function.name` 获取）区分行为。

```python
class CaRRBrowserTool(BaseTool):
    def __init__(self, config: dict, tool_schema: OpenAIFunctionToolSchema):
        super().__init__(config, tool_schema)
        self.tool_server_url = config.get("tool_server_url", "http://localhost:7230")
        self.session_manager = CaRRSessionManager.get_instance()
```

### 3.5.1 create —— 空操作

```python
async def create(self, instance_id=None, **kwargs):
    if instance_id is None:
        instance_id = str(uuid4())
    return instance_id, ToolResponse()
```

**不在 create 中做任何事情**。不 start_session，不初始化状态。因为 ToolAgentLoop 每次调用都 create，而 session 应该只启动一次。

### 3.5.2 execute —— 核心逻辑

```python
async def execute(self, instance_id: str, parameters: dict, **kwargs):
    agent_data = kwargs.get("agent_data")
    if agent_data is None:
        return ToolResponse(text="Missing agent_data"), 0.0, {}

    # 1. 从 agent_data 获取 session 标识和工具参数
    session_id = agent_data.request_id
    create_kwargs = agent_data.tools_kwargs.get(self.name, {}).get("create_kwargs", {})
    search_forbidden_strs = create_kwargs.get("search_forbidden_strs", [])

    # 2. 确保 session 已启动（幂等操作）
    await self.session_manager.ensure_started(
        session_id,
        self.tool_server_url,
        search_forbidden_strs=search_forbidden_strs,
    )

    # 3. browser.open 参数兼容处理
    #    LLM 可能生成 "0"（字符串）而非 0（整数），Tool Server 需要 int
    if self.name == "browser.open" and "id" in parameters:
        try:
            parameters["id"] = int(parameters["id"])
        except (TypeError, ValueError):
            pass    # 如果是 URL 字符串，保持原样

    # 4. 调用 CaRR Tool Server
    remote_env_info = {"search_forbidden_strs": search_forbidden_strs}
    result = await self.session_manager.call_server(
        self.tool_server_url,
        session_id,
        self.name,          # "browser.search" / "browser.open" / "browser.find"
        parameters,         # {"query": "...", "num": 10} / {"id": 0} / {"pattern": "..."}
        remote_env_info,
    )
    return ToolResponse(text=str(result)), 0.0, {}
```

**数据流追踪**——以 `browser.search` 调用为例：

```
LLM 生成: <tool_call>{"name": "browser.search", "arguments": {"query": "breakpoint graph", "num": 10}}</tool_call>
    │
    │ ToolParser 解析
    ▼
FunctionCall(name="browser.search", arguments='{"query":"breakpoint graph","num":10}')
    │
    │ _call_tool → tool.create() → tool.execute()
    ▼
CaRRBrowserTool.execute(parameters={"query":"breakpoint graph","num":10}, agent_data=...)
    │
    │ session_manager.ensure_started("req_001", "http://localhost:7230")
    │ session_manager.call_server("http://localhost:7230", "req_001", "browser.search", ...)
    ▼
HTTP POST http://localhost:7230
{
  "session_id": "req_001",
  "name": "browser.search",
  "arguments": {"query": "breakpoint graph", "num": 10},
  "remote_env_info": {"search_forbidden_strs": ["In the field of..."]}
}
    │
    ▼
CaRR Tool Server → Serper API → Google Search → 过滤 forbidden 结果
    │
    │ 返回 {"output": "[0] Title: ...\n[0] URL Source: ...\n..."}
    ▼
ToolResponse(text="[0] Title: ...\n[0] URL Source: ...\n...")
```

### 3.5.3 release —— 空操作

```python
async def release(self, instance_id: str, **kwargs):
    # 不在这里 close session
    # session 的关闭统一在 CaRRToolAgentLoop.run() 的 finally 中处理
    return
```

### 3.5.4 browser.open 的 id 类型问题

这是一个容易出 bug 的细节。CaRR Tool Server 中：

```python
url = idx2url.get(idx) if isinstance(idx, int) else idx
```

`idx2url` 的 key 是 `int` 类型（0, 1, 2, ...）。如果 LLM 生成 `"id": "0"`（字符串），`isinstance(idx, int)` 为 False，就会被当作 URL 处理，导致错误。

而 verl 的工具 schema（`OpenAIFunctionPropertySchema`）只支持标量类型如 `string`、`integer`，不支持 CaRR 原始的 `"type": ["integer", "string"]`。所以我们在 YAML 中把 `id` 定义为 `string`，然后在 `execute` 中尝试转换：

```python
if self.name == "browser.open" and "id" in parameters:
    try:
        parameters["id"] = int(parameters["id"])
    except (TypeError, ValueError):
        pass    # 如果是 URL，保持字符串
```

```yaml
  # 工具 2: browser.open
  - class_name: "examples.carr_deepsearch.tools.carr_browser_tool.CaRRBrowserTool"
    config:
      type: native
      tool_server_url: "http://localhost:7230"
    tool_schema:
      type: "function"
      function:
        name: "browser.open"
        description: "Open a webpage by search result ID or URL"
        parameters:
          type: "object"
          properties:
            id:
              type: "string"
              description: "ID(integer as string) or URL"
          required: ["id"]

  # 工具 3: browser.find
  - class_name: "examples.carr_deepsearch.tools.carr_browser_tool.CaRRBrowserTool"
    config:
      type: native
      tool_server_url: "http://localhost:7230"
    tool_schema:
      type: "function"
      function:
        name: "browser.find"
        description: "Find a pattern in the currently opened webpage"
        parameters:
          type: "object"
          properties:
            pattern:
              type: "string"
              description: "Pattern to search for"
          required: ["pattern"]
```

**要点**：
- 三个工具使用**同一个类** `CaRRBrowserTool`，通过不同的 `function.name` 区分
- `config.type: native` 表示原生 Python 工具（不是 MCP）
- `config.tool_server_url` 是自定义配置，传入 `__init__` 的 `config` 参数
- `browser.open.id` 使用 `type: "string"` 而不是 `["integer", "string"]`（verl schema 不支持联合类型）

**加载过程**：

```
RL 配置 carr_grpo.yaml 中：
  actor_rollout_ref.rollout.multi_turn.tool_config_path: examples/.../carr_browser_tools.yaml
      │
      │ initialize_tools_from_config()
      ▼
  3 个 CaRRBrowserTool 实例：
  - tools["browser.search"] (name="browser.search")
  - tools["browser.open"]   (name="browser.open")
  - tools["browser.find"]   (name="browser.find")
```

## 3.7 完整工具调用时序

把所有组件串起来，一条完整轨迹的工具调用时序如下：

```
CaRRToolAgentLoop.run(request_id="req_001", ...)
│
├─ 状态: GENERATING
│   LLM 生成 → "browser.search" tool_call
│
├─ 状态: PROCESSING_TOOLS
│   _call_tool("browser.search", ...)
│   │
│   ├── CaRRBrowserTool.create()        → instance_id = uuid
│   ├── CaRRBrowserTool.execute(...)
│   │   ├── session_manager.ensure_started("req_001", ...)
│   │   │   └── HTTP POST: start_session ← 第一次调用，创建 session
│   │   └── HTTP POST: browser.search
│   │       └── Tool Server: sandbox["idx2url"] = {0: url0, 1: url1, ...}
│   └── CaRRBrowserTool.release()       → 空操作
│
├─ 状态: GENERATING
│   LLM 生成 → "browser.open" tool_call
│
├─ 状态: PROCESSING_TOOLS
│   _call_tool("browser.open", ...)
│   │
│   ├── CaRRBrowserTool.create()        → instance_id = 新uuid
│   ├── CaRRBrowserTool.execute(...)
│   │   ├── session_manager.ensure_started("req_001", ...)
│   │   │   └── 已存在，跳过 ← 第二次调用，复用 session
│   │   └── HTTP POST: browser.open
│   │       └── Tool Server: sandbox["cur_web_content"] = "网页内容..."
│   └── CaRRBrowserTool.release()       → 空操作
│
├─ 状态: GENERATING
│   LLM 生成 → "browser.find" tool_call
│
├─ 状态: PROCESSING_TOOLS
│   _call_tool("browser.find", ...)
│   │
│   ├── CaRRBrowserTool.create()
│   ├── CaRRBrowserTool.execute(...)
│   │   ├── ensure_started → 跳过
│   │   └── HTTP POST: browser.find → 在 cur_web_content 中搜索
│   └── CaRRBrowserTool.release()
│
├─ ... (循环多轮) ...
│
├─ 状态: GENERATING
│   LLM 生成最终回答 → 没有 tool_call → TERMINATED
│
└─ finally:
    session_manager.close("req_001", ...)
    └── HTTP POST: close_session → Tool Server 释放 sandbox
```

## 3.8 小结

| 组件 | 职责 | 关键设计 |
|------|------|----------|
| **CaRRSessionManager** | 管理 Tool Server 的 session 生命周期 | 单例模式；request_id 作为 session_id；`ensure_started` 幂等 |
| **CaRRBrowserTool** | 封装三个浏览器操作的 HTTP 调用 | 统一类+不同name；create/release 为空操作；execute 中获取 agent_data |
| **工具 YAML** | 定义三个工具的 schema 和配置 | class_name 指向同一个类；id 用 string 类型 |
| **Session 关闭** | 在 AgentLoop.run() 的 finally 中统一关闭 | 不在 tool.release 中关闭，防止跨轮断裂 |
